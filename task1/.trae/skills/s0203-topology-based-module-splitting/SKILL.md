# s0203 模块拆分与编排

## 作用
- 把已冻结方案和契约拆成可执行的模块边界、依赖 DAG 与并行分组。
- 明确组内并行、组间串行以及失败回退锚点，服务后续实际执行。
- 把"拆分后能不能稳定组织开发"作为主判断，而不是停在目录骨架生成。

## Trigger
- 架构与契约已具备，需要定义模块拆分与执行编排。
- 用户要求做模块划分、职责分配、依赖排序或并行组设计。
- 当前任务若直接开工，会因依赖顺序不清、边界不清而导致并行污染或返工。

## Preconditions
- 已存在一版可核对的最终方案与契约输入，至少能识别核心模块、依赖方向与共享边界。
- 当前任务确实进入"需要组织开发与并行"的阶段；若仍在开放探索或契约未稳，不启动本 Skill。
- 至少可以抽取出初始依赖关系；若连依赖方向都无法判断，先回到上游补契约或补方案。
- 若存在明显循环依赖、边界重叠严重或回退点完全无法定义，直接标记 `未闭合` / `当前不可判定`，不伪造拆分完成。

## TRAE 特化命令
- 若模块边界、优先级、回退范围或并行策略涉及用户偏好与风险容忍度，必须先调用 `AskUserQuestion` 对齐，再输出最终分组。
- 若本 Skill 已输出可执行的模块拆分表、依赖 DAG 或明确阻断说明，应调用 `TodoWrite` 同步当前状态，避免主链误判可以直接并行开工。
- 本 Skill 的拆分与依赖分析必须基于本地方案文本、契约与现有目录语义完成；检索仅使用 `Read` / `Grep` / `Glob` / `SearchCodebase`。

## Context
- 模块拆分必须同时回答职责边界与依赖顺序，不能只给命名列表。
- 并行分组必须附带失败回退边界，不能把失败成本转嫁到执行期。
- 本 Skill 吸收旧 `s402` 的依赖意识，但不回到"步进式合流即万能解"的旧路径。
- 若存在循环依赖或职责重叠，优先暴露结构问题，不允许用目录规划掩盖设计缺陷。
- 模块命名必须采用 `模块N_中文名`，并用数字前缀表达依赖顺序与排序语义。
- `public/` 是共享契约与公共资源边界，模块拆分不得把可共享内容偷塞进业务模块内部。
- 禁止模块间直接导入其他模块的内部实现；跨模块协作只能通过契约、公共资源或显式暴露的稳定边界完成。
- 当前 spec 下，后续执行并行默认最多只放行 2 个 `parallel-sub-agent` 分支；若再细分，也不得突破全局单批并行上限 3。

## Inputs
- 最终方案定稿。
- 已验证契约。
- 依赖线索、共享资源、执行边界、风险约束。
- 可选的旧模块划分稿、目录草案、并行计划草案。

## Outputs
- 模块拆分表。
- 依赖 DAG 或等价的依赖顺序说明。
- 可并行组、组间串行顺序、失败回退锚点。
- 模块命名方案、`public/` 边界说明与跨模块协作约束。
- 下游开发接续入口与阻断说明。

## Action Flow
1. 读取方案与契约，识别核心职责块、共享边界与依赖方向。
2. 形成初版模块拆分表，标明每个模块的职责、输入、输出、依赖以及 `模块N_中文名` 命名。
3. 划清 `public/` 共享边界、模块内部实现边界与禁止跨模块直连内部实现的规则。
4. 构建依赖 DAG 或等价顺序说明，识别哪些可以并行、哪些必须串行，并把当前默认"最多 2 个 `parallel-sub-agent` 分支、全局不超过 3 个并行分支"的上限写入编排结果。调度约束：
   - **subagent_type 显式要求**：每个涉及 subagent 调度的并行组必须显式写明 `subagent_type`（`parallel-sub-agent` / `general_purpose_task`），不得留给执行期临场判断。
   - **actual agent id 回填**：启动 subagent 后必须立即将真实 agent id 回填至台账，失败回退点同步记录。
   - **未保护上下文隔离**：若并行分支涉及未保护上下文（测试意图、审查中间态、未闭合判断等），必须按 `rules-0 §四-12` 隔离启动，不得将当前未闭合上下文直接分发给子分支。
5. 若边界划分、并行策略或回退范围存在真实歧义，调用 `AskUserQuestion` 对齐。
6. 输出并行组、串行顺序、失败回退锚点与开发接续入口。
7. 调用 `TodoWrite` 回填当前处于"可组织执行"还是"仍不可判定"。

## hc_sc_ec_mapping
- 主映射：`HC-2` `SC-2` `EC-2`
- 次映射：`HC-3` `EC-1` `EC-6` `EC-7`
- 说明：
  - `HC-2`：把整体方案分解成可执行、可衔接的模块单元。
  - `SC-2`：显式区分组内并行与组间串行，是本 Skill 的核心价值。
  - `EC-2`：并行分支必须边界清晰，避免失败相互污染。
  - `HC-3` / `EC-1`：必须交代当前拆到了哪里、失败时退回哪里。
  - `EC-6/EC-7`：当边界或依赖无法安全裁决时，必须解释并请示。

## State Protocol
- `已闭合`：模块拆分、依赖顺序、并行分组与回退锚点都已明确，可组织后续执行。
- `未闭合`：拆分已形成初稿，但依赖冲突、边界重叠或回退范围仍不完整，不能稳定下推。
- `当前不可判定`：当前输入不足以可靠定义模块拆分与依赖编排，或关键依赖关系仍无法判断。

## Closure Signals
- 存在一份覆盖职责、输入、输出与依赖的模块拆分表。
- 模块命名方案符合 `模块N_中文名`，且共享资源与 `public/` 边界已被显式划清。
- 明确给出依赖 DAG 或等价顺序说明，而不是只列模块名称。
- 明确给出并行组、串行顺序与失败回退锚点。
- 并行编排结果显式写出当前默认最多 2 个 `parallel-sub-agent` 分支，且任何细分不突破全局上限 3。
- 若状态不是 `已闭合`，必须说明是边界问题、依赖问题还是回退设计问题。

## hc3_handoff
- 做到哪里：已形成可执行的模块拆分与编排结果，或明确说明当前为什么还不能安全组织执行。
- 为什么做到这里：因为真正开工前，必须先回答谁能并行、谁必须等待、失败时如何局部回退。
- 哪些已闭合：已确认的模块职责、已明确的依赖方向、已锁定的并行组与回退边界。
- 哪些未闭合：循环依赖、职责重叠、尚未定稿的共享边界、无法确认的串并行顺序。
- 下一步从哪接：
  - 若 `已闭合`，进入后续模块开发与执行编排
  - 若非 `已闭合`，先补契约、重构边界或回退到上游方案节点

## ec7_trigger_gate
- 模块边界或依赖方向存在关键歧义。
- 并行组划分会明显影响风险暴露与返工成本，且当前无法自动裁定。
- 用户要求立即开工，但当前拆分结果仍会导致跨分支污染或回退失效。

## ec6_minimum_explanation
- 当前为什么可以或不可以把拆分结果作为正式执行编排输入。
- 已经确定了哪些模块、依赖和并行组，哪些仍有冲突。
- 已做了哪些依赖分析与回退设计。
- 建议下一步补哪类边界或裁决，才能继续执行。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否被定位为"模块拆分与编排器"而非普通目录生成器。
- 核心段审查：检查模块拆分、依赖 DAG、并行组与回退锚点是否齐备。
- 执行阶段：检查执行者是否真的按既定依赖顺序组织并行，并在分支失败时局部回退，而不是临场拍脑袋重排。
- 交付前审查：检查正式目录中的模块拆分 Skill 是否仍能支撑实际开发组织，并与最终契约保持一致。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"模块拆分 / 依赖 DAG / 并行分组 / 串行顺序"类触发语义。
- 首步动作明确为"读取方案与契约并识别职责块与依赖方向"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0203-topology-based-module-splitting`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的模块拆分 / 依赖 DAG / 并行分组语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"读取方案与契约并识别职责块与依赖方向"启动，以及阻断点。
- `依赖与并行核对`：记录模块命名、`public/` 边界、依赖顺序、回退锚点，以及"默认最多 2 个 `parallel-sub-agent`、全局不超过 3"是否被显式写入。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录依赖冲突、循环依赖、并行缩减理由与后续接续入口。

## fallback_and_escalation
- 若存在循环依赖或职责重叠，先暴露结构问题，不伪造拆分完成。
- 若并行组回退边界不清，先缩小并行范围或请示，不让失败污染整体执行。
- 若拆分结果要求超过 2 个 `parallel-sub-agent` 才能成立，或任何设计突破全局单批上限 3，必须先降级为更小并行组或串行，再决定是否请示。
- 若模块命名、`public/` 边界或跨模块导入约束无法同时成立，优先回退拆分方案，不拿目录骨架冒充编排完成。
- `EC-4` 间接补偿：由 GN-004、依赖分析与主 agent 复核防止编排偏航。
- `EC-5` 间接补偿：若当前能力不足以可靠判断复杂依赖或回退范围，应升级请示，而不是输出看似完整的分组表。
