# s0401 安全文件写入

## 作用
- 作为 AC 范式高危资产的专属防火墙，拦截会触碰核心契约、规则中枢和大范围破坏性操作的写盘动作。
- 在执行层承接 `ec7_action_gate` 与高危路径请示逻辑，避免 AI 用无语义文件工具顺手改坏真相源。
- 对工程交接锚点做格式与留痕兜底，但不把普通业务开发写成层层请示的绊脚石。

## Trigger
以下任一条满足即**必须**触发本 Skill，不可因人类已授权而跳过：
- 任务需要修改 `public/`、`.trae/rules/`、全局 `AGENTS.md`、入口文件或其他 AC 范式核心资产。
- 任务涉及跨越 2 个以上模块的批量覆盖、删除、移动或重构。
- 任务需要写入工程交接锚点，如 `.trae/documents/`、spec 三件套、`current-note.md`，且需要校验命名、元数据或格式合规性。
- 若只是 `modules/`、`tests/`、`global_mock/` 的日常业务开发、测试编写或普通局部修补，默认不触发本 Skill。

## Preconditions
- 已明确目标路径、操作类型与改动范围，而不是泛泛地“帮我改一下”。
- 已能判断当前目标属于以下哪一类：免检通行区 / 核心资产 / 跨模块破坏性动作 / 工程交接锚点。
- 若目标属于核心资产或破坏性动作，必须先判断这是“直接阻断”还是“请示后再决定”，而不是默认进入写盘流程。
- 若目标属于工程交接锚点，必须具备可核对的格式规则（如命名规范、frontmatter、必填章节）。
- 若当前动作只是普通业务开发写盘，不应进入本 Skill；应直接回到常规开发与测试流程。

## TRAE 特化命令
- 若用户只给模糊改动意图、未指定写入范围或越权边界不清，必须先调用 `AskUserQuestion` 做结构化澄清。
- 若本 Skill 已完成一次安全写入或明确阻断说明，应调用 `TodoWrite` 同步真实状态与当前回退点。
- 本 Skill 的文件定位与内容核对只允许使用 `Read` / `Grep` / `Glob` / `SearchCodebase` 做本地检索；实际编辑使用安全写入工具，不依赖临时 shell 脚本批量覆盖。

## Context
- 本 Skill 负责的是"高危写盘判定与引流"，不是通用文件 IO 助手。
- 免检通行区默认包括 `modules/`、`tests/`、`global_mock/` 等日常业务开发区；这些区域依赖版本控制与常规测试兜底，不在本 Skill 内层层请示。
- 高危区默认包括 `public/`、`.trae/rules/`、全局 `AGENTS.md`、入口文件，以及跨模块的大范围破坏性操作。
- 工程交接锚点（如 `.trae/documents/`、spec 三件套、`current-note.md`）允许频繁写入，但必须格式合规、留痕可追溯。
- 本 Skill 的主问题是"是否允许继续动作、是否需要请示、是否需要改走其他流程"，而不是亲自执行复杂补丁写入。

## Inputs
- 目标文件绝对路径与操作类型。
- 计划写入内容或动作摘要。
- 当前任务边界、受影响模块范围与是否涉及核心资产。
- 若写入工程交接锚点，对应的命名规范、元数据要求或章节模板。

## Outputs
- 判定结果：`ALLOWED / ASK_USER / BLOCKED / REWRITE_REQUIRED`。
- 场景归类：`免检通行区 / 核心资产 / 破坏性动作 / 工程交接锚点`。
- 路径与影响面判断结果。
- 若阻断，给出应改走的流程或请示原因。
- 若为工程交接锚点，给出格式校验结果与需补项。
- 三值状态、未闭合项与后续接续建议。

## Action Flow
1. 查验目标路径与动作类型，先判断它属于哪类场景：免检通行区 / 核心资产 / 跨模块破坏性动作 / 工程交接锚点。
2. 若属于免检通行区，直接输出 `ALLOWED`，把控制权交还给常规写盘与测试流程；不得把普通业务开发伪装成高风险写入。
3. 若属于核心资产（如 `public/`、`.trae/rules/`、全局 `AGENTS.md`、入口文件），直接阻断并说明这是 AC 范式真相源；需要改动时应改走专门流程（如契约变更走 `s0601`）或由人类显式处理。
4. 若属于跨模块破坏性动作，先罗列影响面、涉及模块和动作类型，再调用 `AskUserQuestion` 构造选择题请示是否授权执行；未获人类裁决前保持挂起。
5. 若属于工程交接锚点，执行格式与留痕校验：检查命名规范、frontmatter、必填章节、关联文件与状态字段是否齐备；不合规时输出 `REWRITE_REQUIRED` 并要求先修正文档本身。
6. 对被判定为允许继续的场景，只输出放行结论、格式校验结果或升级路径，不在本 Skill 内承担复杂补丁写入。
7. 输出场景归类、判定结果、影响面/格式检查结论、三值状态与后续接续建议，并调用 `TodoWrite` 回填状态。

## hc_sc_ec_mapping
- 主映射：`HC-1` `EC-1` `EC-2`
- 次映射：`HC-3` `SC-3` `EC-6` `EC-7`
- 说明：
  - `HC-1`：先判断当前写入动作是否真的可以闭合，而不是默认一写了之。
  - `EC-1`：任何写入都必须带回滚锚点，失败后能回到已知有效状态。
  - `EC-2`：通过路径边界、最小补丁和原子写入限制错误传播半径。
  - `HC-3`：交代改到哪里、为什么停、回退到哪、下一步从哪接。
  - `SC-3`：越权或未留痕时不得假装写入成功。
  - `EC-6/EC-7`：边界不明或路径高风险时必须解释并请示。

## State Protocol
- `已闭合`：当前判定已完成，且后续动作路径明确（放行 / 阻断 / 请示 / 重写文档）；
- `未闭合`：当前动作已识别为高危或不合规，但仍待人类裁决、格式修正或改走其他流程；
- `当前不可判定`：当前无法判定目标路径性质、影响面或格式规则，不能继续动作。

## Closure Signals
- 已明确当前动作属于哪类场景，而不是笼统说“这是写文件”。
- 若为免检通行区，已明确允许继续且不升级请示。
- 若为核心资产或破坏性动作，已明确阻断或请示原因。
- 若为工程交接锚点，已明确格式是否合规、缺了什么。
- 明确给出三值状态，而不是只说“已处理”。

## hc3_handoff
- 做到哪里：已完成放行判定、已阻断高危动作、已发起请示，或已要求重写不合规锚点文档。
- 为什么做到这里：因为本 Skill 的职责是守住 AC 范式高危资产下限，而不是接管所有写盘动作。
- 哪些已闭合：已明确归类的场景、已完成的影响面或格式校验、已形成的放行/阻断/请示结论。
- 哪些未闭合：仍待人类授权的破坏性动作、仍待修正的文档格式、仍待澄清的路径归类。
- 下一步从哪接：
  - 若 `已闭合` 且为 `ALLOWED`，回到常规写盘与测试流程
  - 若为 `BLOCKED`，改走专门流程或等待人类处理
  - 若为 `ASK_USER` 或 `REWRITE_REQUIRED`，先完成请示或修正文档后再尝试

## ec7_trigger_gate
- 写入目标落在 `public/`、`.trae/rules/`、全局 `AGENTS.md`、入口文件或其他核心真相源。
- 动作涉及跨越 2 个以上模块的批量覆盖、删除、移动或重构。
- 用户意图、任务边界或允许范围不清，无法判断当前是否属于高危破坏性动作。
- 任务试图绕过专门流程直接改契约、规则或公共资产。

## ec6_minimum_explanation
- 当前动作属于哪类场景，为什么这样归类。
- 为什么允许继续、必须请示、必须阻断，或必须先重写文档。
- 已核对哪些路径边界、影响面或格式规则。
- 若被阻断，应改走哪个流程或补什么授权。
- 若文档不合规，应补哪些命名、元数据或章节后再继续。

## gn004_touchpoints
- 三件套阶段：检查该 Skill 是否被收窄为 AC 范式高危资产防火墙，而非广义写盘助手。
- 关键审查点阶段：检查免检通行区、高危区、工程交接锚点三类边界是否都进入正文。
- 执行阶段：检查执行者是否真的把普通开发放行，把高危动作阻断或请示，而不是对所有写盘一刀切升级。
- 交付前审查：检查最终结论是否仍保持“高危防守底线”，且没有污染普通业务开发心流。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"public/ / .trae/rules/ / AGENTS.md / 入口文件 / 批量删除 / 批量覆盖 / .trae/documents / current-note"这类高危防火墙语义。
- 首步动作明确为"判定场景归类并放行或拦截"，主执行者可独立执行。
- 正文能同时看出"普通开发严禁触发"与"三大高危场景必须触发"这两条边界。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0401-safe-file-writing`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的高危场景语义、用户语言样例、未命中原因或 `当前不可判定` 理由。
- `首步独立执行`：记录是否能直接从"判定场景归类并放行或拦截"启动，以及阻断点。
- `场景归类判定`：记录当前动作属于免检通行区 / 核心资产 / 破坏性动作 / 工程交接锚点中的哪一类。
- `闸门结果核对`：记录最终是 `ALLOWED / ASK_USER / BLOCKED / REWRITE_REQUIRED`，以及原因。
- `是否误触发`：记录普通 `modules/` / `tests/` / `global_mock/` 开发是否被错误拉入本 Skill。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录影响文件、回滚情况、越权项、未闭合项与后续接续入口。

## fallback_and_escalation
- 若目标属于免检通行区，直接放行，不在本 Skill 内追加多余请示。
- 若目标属于核心资产，直接阻断并引导到专门流程或人类操作，不尝试灰色写入。
- 若目标属于跨模块破坏性动作，先请示，不得“先做一部分再说”。
- 若目标属于工程交接锚点且格式不合规，先重写文档本身，再继续后续流程。
- 若路径归类仍不清楚，保持 `当前不可判定`，不要硬套到某个场景里继续动作。
- `EC-4` 间接补偿：由 GN-004、问题留痕规则与主 agent 复核防止执行偏航或越权扩散。
- `EC-5` 间接补偿：若当前能力不足以判断权限、影响面或恢复策略，应升级请示，而不是硬写盘。
