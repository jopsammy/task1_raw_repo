# s0601 契约变更适配

## 作用
- 在契约发生变化后，识别受影响范围、同步边界与必须阻断的下游动作。
- 把契约变更后的适配任务拆成可追踪、可留痕、可审查的接续清单，而不是假装一键完成全链路更新。
- 把"哪些资产现在必须同步、哪些开发分支必须暂停"作为主问题，而不是只生成一份差异说明。

## Trigger
- 用户要求修改契约、更新 schema、调整接口存根、修改配置契约或重生成 Mock。
- 当前已确认 `public/schema/`、`public/interface_stub/`、`public/config_template/` 对应契约发生真实变化。
- 下游已有 Mock、模块拆分、规则模板、测试入口或实施分支依赖旧契约，继续推进会出现边界漂移。

## Preconditions
- 已存在可读取的契约变更输入，包括旧版与新版契约，或至少存在可靠的差异说明。
- 当前确实发生了"契约真相源变化"，而不是单纯讨论变更可能性。
- 已能识别至少一类下游依赖：Mock、规则模板、模块拆分、测试入口、实现分支或交付说明。
- 若无法确认变更是否已被批准、旧新契约差异是否真实存在，先保持 `未闭合` 或 `当前不可判定`，不伪造适配完成。
- 若当前其实是第一次冻结契约、尚不存在旧新差异，则回退到 `s0201`，不把"首轮定稿"误写成"变更适配"。

## TRAE 特化命令
- 若变更是否被批准、影响范围优先级或适配顺序存在真实歧义，必须先调用 `AskUserQuestion` 做结构化对齐，再输出适配计划。
- 若本 Skill 已形成影响面清单、阻断结论或适配接续计划，应调用 `TodoWrite` 同步真实状态，避免主链误判"契约改完就等于全链闭合"。
- 本 Skill 的核对只能依赖本地契约、既有草稿、规则文件、测试资产与当前上下文完成；检索仅使用 `Read` / `Grep` / `Glob` / `SearchCodebase`。

## Context
- 本 Skill 承接的是"契约变更后的影响分析与同步编排"，不替代对 `public/` 的随意修改，也不越权直接替代所有下游实现。
- 契约一旦变化，至少要重新检查三层契约一致性、Mock 依赖、模块拆分、规则模板和相关验证入口是否仍成立。
- 对 `public/` 的修改必须视为已发生或已批准前提下的读取对象；本 Skill 自身不以"直接写改 public 契约"为成功标准。
- 若影响面尚未厘清，必须先阻断继续开发或交付，而不是让各分支自行吸收变更。
- 适配输出必须能支持后续局部回退、重新审查与 GN-004 抽查，不能只是一份模糊待办。
- 适配接续清单未完整形成前（差异摘要、影响面分级、阻断项、同步顺序、回退锚点任一缺失），不得向人类宣告已完成。半成品不得汇报。

## Inputs
- 旧版与新版契约，或等价的差异说明。
- 依赖这些契约的 Mock、规则模板、模块拆分结果、测试入口与相关实现线索。
- 当前任务状态、活跃分支、未完成事项与可能受影响的审查节点。
- 可选的变更批准信息、迁移窗口或风险优先级说明。

## Outputs
- 契约差异摘要与影响面分级结果。
- 受影响资产清单：至少覆盖 Mock、规则模板、模块拆分、测试/验证入口、实现分支。
- 阻断项、可继续项与必须重新审查的节点。
- 适配接续清单：谁先同步、谁后同步、失败时回退到哪里。
- 结构化 note 留痕：将适配结论写入 `current-note.md` 或 `.trae/documents/` 变更记录。
- 三值状态、未闭合项与后续请示入口。

## Action Flow
1. 读取旧版与新版契约，确认变更是否真实成立，并拆出字段、签名、异常、默认值与配置责任的差异。
2. 沿着当前矩阵依赖链检查下游影响面，至少覆盖 `s0202`、`s0203`、`s0301`、测试入口与正在推进的实现分支。
3. 按"必须立即阻断 / 必须同步更新 / 可延后复核"三类分级整理影响面，并识别哪些节点需要重新提醒主线程拉起 GN-004 审查。
4. 若影响面涉及高风险分支、交付节点或多条并行链路，且当前无法自动裁定同步顺序，必须先调用 `AskUserQuestion` 对齐，不得在未获人类裁决前继续输出适配清单。
5. 若变更优先级、批准状态或同步顺序存在真实歧义，调用 `AskUserQuestion` 对齐。
6. 输出适配接续清单：先同步什么、如何重跑验证、哪些分支应暂停、失败时回退到哪个已知有效锚点。
7. 适配接续清单未完整形成前（至少包含差异摘要、影响面分级、阻断项、同步顺序、回退锚点），不得向人类宣告"已完成"。若任何必要字段缺失，必须先阻断并请示，不得以"后续补全"为由先行汇报。
8. 将适配结论以结构化 note 写入当前 spec 的 `current-note.md`（若当前无活跃 spec，则写入 `.trae/documents/` 下新建变更留痕），至少记录：变更摘要、受影响资产、阻断项与可继续项、同步顺序、回退锚点、未闭合项、接续入口。
9. 若适配接续清单已闭合，必须显式提醒主线程：请主线程拉起 GN-004 对本适配结论进行独立审查。
10. 调用 `TodoWrite` 回填当前处于"影响面已冻结可执行"还是"仍不可判定/仍待批准"。

## hc_sc_ec_mapping
- 主映射：`HC-2` `SC-2` `EC-2`
- 次映射：`HC-3` `EC-1` `EC-3` `EC-6` `EC-7`
- 说明：
  - `HC-2`：把契约变更拆成可执行的同步子任务，而不是整体恐慌式返工。
  - `SC-2`：明确哪些依赖必须先同步、哪些分支需要暂停，是本 Skill 的核心价值。
  - `EC-2`：通过影响面分级限制变更污染半径，避免一处契约变化把全链拖崩。
  - `HC-3` / `EC-1`：必须说明当前变更已处理到哪里、失败时退回哪里。
  - `EC-3`：适配结论应可被 GN-004 或后续执行者独立核对，而不是只听执行者描述。
  - `EC-6/EC-7`：批准状态不清、风险过高或同步顺序无法裁定时必须解释并请示。

## State Protocol
- `已闭合`：影响面、阻断项、同步顺序与回退锚点都已明确，可进入后续局部适配执行。
- `未闭合`：差异已识别，但影响面、批准状态、同步顺序或回退设计仍不完整，不能安全下推。
- `当前不可判定`：当前无法确认契约是否真实变更、影响面是否可核对，或关键依赖状态不足以形成可靠判断。

## Closure Signals
- 存在可核对的旧新契约差异摘要，而不是只说"契约改了"。
- 已列出至少 Mock、规则模板、模块拆分、测试入口与实现分支五类影响面中的相关项。
- 明确给出阻断项、可继续项、重新审查节点与同步顺序。
- 明确给出失败回退锚点与后续接续入口。
- 若状态不是 `已闭合`，必须说明当前卡在批准、差异确认、影响面识别还是回退设计。

## hc3_handoff
- 做到哪里：已完成影响面识别与接续拆解，或明确说明当前为什么无法安全适配。
- 为什么做到这里：因为契约一旦变化，继续沿用旧边界会让 Mock、模块、规则与实现全部产生静默漂移。
- 哪些已闭合：已确认的差异项、已冻结的影响面分级、已锁定的阻断节点与同步顺序。
- 哪些未闭合：未批准的变更、未核清的依赖、尚未决定的迁移窗口或回退路径。
- 下一步从哪接：
  - 若 `已闭合`，进入局部适配执行、重跑验证或重审节点
  - 若非 `已闭合`，先补批准、缩小影响面或回退到上游契约裁决

## ec7_trigger_gate
- 无法确认契约变更是否已被批准或是否真实生效。
- 影响面涉及高风险分支、交付节点或多条并行链路，当前无法自动决定同步顺序。
- 影响面涉及多条并行链路且无法自动决定同步顺序时，必须先调用 `AskUserQuestion` 请示，不得自行裁定。
- 用户要求直接继续开发，但现有下游资产明显仍依赖旧契约。

## ec6_minimum_explanation
- 当前为什么可以或不可以进入契约变更后的适配执行。
- 已识别哪些差异和受影响资产，哪些仍待核清。
- 已完成哪些阻断判断、依赖分级与回退设计。
- 建议下一步先同步什么、暂停什么，或需要谁批准后再继续。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否被定义为"变更影响面与同步入口"，而不是重型自动编排器。
- 执行阶段：检查执行者是否真的根据影响面清单局部同步、局部阻断，而不是让各分支自行吸收变更。
- 关键检查点阶段：当契约变更影响到核心边界、并行顺序或交付口径时，应提醒主线程拉起 GN-004 抽查。
- 交付前审查：检查所有受影响节点是否已按清单完成同步或显式保留未闭合项。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"契约变更 / schema 更新 / 接口变更 / 影响面适配 / Mock 重生成"类触发语义。
- 首步动作明确为"读取旧新契约并识别差异与受影响下游"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0601-adapting-contract-changes`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的契约变更 / 影响面适配语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"读取旧新契约并识别差异与受影响下游"启动，以及阻断点。
- `适配覆盖核对`：记录差异摘要、影响面分级、阻断项、同步顺序、回退锚点、重审节点是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录引用契约、受影响资产、批准状态、未闭合项与接续入口。

## fallback_and_escalation
- 若契约变更尚未批准或差异不真实，先阻断，不伪造适配完成。
- 若影响面过大且当前无法一次厘清，先缩到"必须立即阻断"的最小集合，再逐批扩展，不做一口气全链改造。
- 若发现某些下游资产仍依赖旧契约但短期无法同步，必须显式列为阻断项或已知风险，不允许静默跳过。
- `EC-4` 间接补偿：由 GN-004、契约一致性核对与主 agent 复核防止执行者把"变更已知晓"误写成"变更已适配"。
- `EC-5` 间接补偿：若当前能力不足以裁定复杂影响面、迁移窗口或同步顺序，应升级请示，而不是输出看似完整的全链适配结论。
