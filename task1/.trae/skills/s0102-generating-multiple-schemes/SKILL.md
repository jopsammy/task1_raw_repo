# s0102 多方案生成

## 作用
- 通过 parallel-sub-agent 在隔离上下文中并行制造至少 3 条真实候选方案，防止单一上下文线性脑补。
- 为 `s0103` 提供可融合、可比较、可审查的输入，而不是表面差异。
- 把"方案分叉是否真实成立"作为重点检查对象。

## Trigger
- 结构化需求已形成，需要进入方案设计。
- 用户要求出多个候选方案、做架构路线对比、做保守 / 平衡 / 激进探索。
- 当前任务若直接进入单方案，会导致过早收缩与假确定性。

## Preconditions
- `s0101` 已至少产出一版可读的结构化需求，或当前已有等价的需求收束结果可供消费。
- 当前任务确实存在多路径设计价值；若目标天然单解、约束已锁死到没有真实分叉空间，不启动本 Skill。
- 至少能识别一组分叉维度，例如风险、成本、速度、复杂度、扩展性中的一种以上；否则先补约束或请示。
- 若输入仍不足以支撑真实分叉，必须输出 `当前不可判定` 或回到上游补信息，不得硬造三套空方案。
- `parallel-sub-agent` 可用：若不可用，按降级路径执行（见 fallback_and_escalation）。

## TRAE 特化命令
- 若"保守 / 平衡 / 激进"之外还存在多种有效分叉口径，且需要用户明确偏好，先调用 `AskUserQuestion` 选择分叉维度，再生成方案。
- 若该 Skill 已经形成真实可比较的多方案结果，应调用 `TodoWrite` 同步当前状态，避免主链误判为"还没开始"或"已经定稿"。
- 本 Skill 的本地资料检索仅使用 `Read` / `Grep` / `Glob` / `SearchCodebase`；不得把"方案分叉"工作降格为非本地检索代理的搜索汇总。

## Context
- **方案必须是实质分叉，而不是措辞差异。** 同一上下文中线性脑补的多方案（如"方案A：激进一点 / 方案B：保守一点"的措辞变化）不视为真实分叉。
- 真实分叉必须满足：每条方案由独立的 parallel-sub-agent 在隔离上下文中生成，各自持有不同的设计倾向。
- 至少形成 3 条候选路线，且每条路线都能对应到结构化需求。
- 输出必须直接服务 `s0103`，不能只给零散脑暴点子。
- 输入若不充分，应优先暴露"不能真实分叉"的原因。
- 并发上限遵循 `rules-0 §四-4`：单批并行 ≤2，全局并行 ≤3。若需 ≥4 个 subagent，分两批串行启动。

## Inputs
- `s0101` 产出的结构化需求。
- 可选的技术栈偏好、资源约束、时间边界。
- 可选的历史方案或禁用项。

## Outputs
- 至少 3 条由独立 subagent 生成的真实分叉方案。
- 每条方案附带生成该方案时采用的设计倾向说明。
- 覆盖度检查、风险表、适用场景表。
- 差异表，为 `s0103` 提供融合输入。
- 若 parallel-sub-agent 不可用，输出降级路径说明。

## Action Flow
1. **分叉维度确认**：读取结构化需求，锁定不可违背的核心约束。列出可能存在的有意义分叉维度（最少 3 个，如保守/平衡/激进，或成本优先/速度优先/可靠性优先）。若维度选择存在真实歧义或反直觉组合，调用 `AskUserQuestion` 请示人类是否认可这些发散方向。
2. **人类认可**：等待人类对分叉维度做出裁决。不得在人类裁决前拉取 subagent。
3. **隔离并行分叉**：人类认可后，通过 `parallel-sub-agent` 在隔离上下文中并行生成方案。每个 subagent 必须被指派一个明确的设计倾向（如"保守方案""激进方案"），各 subagent 之间不存在上下文污染。

   调用方式：
   ```
   launch(subagent_type="parallel-sub-agent", task="基于[结构化需求]，以[设计倾向]生成一个完整方案。...")
   ```
   并发约束：
   - 单批 ≤ `MAX_PARALLEL_PER_BATCH`（2），全局 ≤ `MAX_PARALLEL_GLOBAL`（3）
   - 若需要 ≥4 个 subagent，分两批：第1批启动2个，闭合后启动第2批
   - 启动后立即将 `actual agent id` 回填台账

4. **汇报**：所有 subagent 完成后，汇总各方案。对每条方案做需求覆盖度和风险扫描。输出差异表，标注各方案的倾向性差异是否真实成立。
5. **融合入口请示**：调用 `AskUserQuestion` 向人类汇报方案概况与差异表，请示"是否允许开始考虑融合方案"。不得在人类确认前直接进入 s0103。
6. 调用 `TodoWrite` 回填该节点已处于"可进入融合 / 未闭合 / 当前不可判定"。
7. **提醒主线程**：若本 Skill 已产出闭合结果且准备交接给 s0103，必须显式提醒主线程：请主线程在交付前适时拉起 GN-004 对本 Skill 的多方案质量进行独立审查。

## hc_sc_ec_mapping
- 主映射：`SC-2` `HC-2` `EC-3`
- 次映射：`SC-3` `EC-6` `EC-7`
- 说明：
  - `SC-2`：本 Skill 的核心职责就是通过隔离上下文制造真实决策分叉。
  - `HC-2`：分叉必须形成后续可审查、可收束的子输入。
  - `EC-3`：需要接受 GN-004 对"伪多方案"的独立审查。
  - `SC-3`：若输入不足或 subagent 不可用，不得把伪分叉标成已完成。
  - `EC-6/EC-7`：分叉维度不清或决策依据不足时必须解释并请示。

## State Protocol
- `已闭合`：至少 3 条由独立 subagent 生成的真实分叉方案已形成，且均有覆盖度与风险说明。
- `未闭合`：已形成部分方案，但差异不足或输入不完整，仍不能进入融合。
- `当前不可判定`：输入不足以支持真实分叉，或 parallel-sub-agent 不可用且降级路径不适用。

## Closure Signals
- 至少存在 3 条非同质化方案，每条由独立 subagent 在隔离上下文中生成。
- 每条方案均有核心思路、适用场景、风险点、生成时所持的设计倾向。
- 存在一份差异表，能够支持下游融合与审查。
- 未闭合或不可判定时，必须明确说明缺了什么。
- subagent 台账中 `actual agent id` 已回填。

## hc3_handoff
- 做到哪里：已经把需求扩展成多个隔离并行的候选路径，或明确说明为什么当前做不到。
- 为什么做到这里：因为后续融合必须建立在真实分叉而非单解幻觉上。
- 哪些已闭合：已形成的方案及其差异说明、各方案的设计倾向。
- 哪些未闭合：差异不足的分叉、缺少关键约束支撑的方案。
- 下一步从哪接：
  - 若 `已闭合`，进入 `s0103`
  - 若非 `已闭合`，先补信息或回到 `s0101`

## ec7_trigger_gate
- 仅能形成同质化方案，无法制造真实分叉 → 必须请示人类：是接受弱分叉继续，还是回退。
- 关键约束缺失导致分叉依据不足 → 请示人类补约束。
- 用户要求直接进入融合，但当前候选方案差异不真实 → 阻断并请示。
- **无法制造真实分叉时必须请示**（新增）：若 parallel-sub-agent 不可用或无法达到 ≥3 个隔离方案，必须转型为 AskUserQuestion 并说明不可用原因与替代路径。
- 关键缺口存在但未拉起 AskUserQuestion → 必须阻断。

## ec6_minimum_explanation
- 当前为什么可以或不可以生成多方案。
- 已形成哪些候选路线，差异在哪里，各方案由哪个 subagent（agent id）生成。
- 哪些约束支撑分叉，哪些仍缺失。
- 建议下一步交由 `s0103` 融合，或回退补信息。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否真的要求"通过 parallel-sub-agent 强制隔离并行多方案"，而不是软建议。
- 核心段审查：检查是否存在伪分叉、伪差异或凑数方案；检查 subagent 台账是否齐备。
- 执行阶段：检查执行者是否真的通过独立 subagent 产出 3 条非同质化方案，而不是单上下文轻改三份表面不同。
- 交付前审查：检查其输出是否足以支撑 `s0103` 的融合。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"多方案 / 架构方案 / 真实分叉 / 隔离并行 / 方案对比 / parallel-sub-agent"等触发语义。
- 首步动作明确为"读取结构化需求并列出分叉维度供人类确认"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0102-generating-multiple-schemes`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的多方案 / 架构方案 / 真实分叉 / 隔离并行语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"读取结构化需求并列出分叉维度供人类确认"启动，以及阻断点。
- `分叉质量核对`：记录方案数量、同质化检查、各方案 subagent agent id、覆盖度对比、差异表与下游融合入口是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录方案差异项、缺失维度、约束缺口、agent id 台账与后续接续入口。

## fallback_and_escalation
- 若输入不足以支撑真实分叉，先输出阻断说明，不硬造三套空方案。
- 若方案差异不足，先重做分叉逻辑，不把措辞差异冒充多方案。
- **parallel-sub-agent 不可用时降级**（新增）：若 `parallel-sub-agent` 不可用：
  1. 通过 AskUserQuestion 告知人类"parallel-sub-agent 不可用，无法实现真实隔离分叉"
  2. 提供两个选项：① 降级为 `general_purpose_task` 串行生成方案（接受无隔离上下文风险）② 暂停，等待 parallel-sub-agent 恢复
  3. 若用户选择降级路径，在 output 中标注"非隔离生成——方案之间可能存在上下文污染"
  4. 台账中记录降级事实
- `EC-4` 间接补偿：由 GN-004、差异检查与主 agent 复核防止伪分叉。
- `EC-5` 间接补偿：遇到无法自动决定的分叉维度时升级请示，而不是自选一套凑数。
