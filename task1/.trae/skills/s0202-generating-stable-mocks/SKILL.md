# s0202 稳定 Mock 生成

## 作用
- 基于已验证契约生成可供并行开发直接使用的稳定 Mock 依赖。
- 明确 Mock 的策略、替换边界与验证入口，避免其演化成新的漂移源。
- 把"是否能稳定支撑并行"作为主问题，而不是只产出一份能跑的替身。

## Trigger
- 接口契约已经通过，需要为并行开发提供稳定替身。
- 用户要求生成 Mock、测试替身、预生成数据或开发期依赖桩。
- 当前若等待真实实现会阻塞后续开发，但又不能容忍随意假设接口行为。

## Preconditions
- `s0201` 已产出一版可读且可核对的契约结果，或当前已有等价的已验证契约输入。
- 至少可以识别 Mock 需要覆盖的核心接口、关键字段与典型响应场景；若契约仍过于空泛，不启动本 Skill。
- 当前任务确实处于"需要稳定替身支持并行"的阶段；若真实实现已就绪或无需并行，不启动本 Skill。
- 若契约未通过、一致性未确认或替换边界完全不清，直接阻断或标记 `当前不可判定`，不输出伪稳定 Mock。
- 若当前问题是"旧新契约已经变化，需判断哪些 Mock 必须重生成、哪些分支必须暂停"，优先转入 `s0601`，不把变更适配伪装成首轮稳定 Mock 生成。

## TRAE 特化命令
- 若 Mock 的保真度、覆盖范围、切换边界或失败策略涉及用户偏好与风险容忍度，必须先调用 `AskUserQuestion` 对齐，再生成稳定 Mock。
- 若本 Skill 已产出可交付 Mock 集、替换说明或阻断结论，应调用 `TodoWrite` 同步当前状态，避免主链误判 Mock 已可用。
- Mock 生成所依赖的信息必须来自本地契约文本、现有接口定义与当前上下文；检索仅使用 `Read` / `Grep` / `Glob` / `SearchCodebase`。

## Context
- Mock 依附于契约，不替代真实实现。
- 输出必须同时包含 Mock 结果、验证信号与替换边界。
- 本 Skill 的目标是稳定支撑并行开发，而不是追求"看起来像真系统"。
- 若 Mock 与契约冲突，应优先阻断，不允许让下游带着假依赖继续开发。
- 默认稳定依赖应落到 `public/pre_generated_mock/`，用于为所有分支提供统一默认替身。
- `public/global_mock/` 只用于显式覆盖默认 Mock 的特殊测试场景，不能被当作绕过契约的兜底黑盒。
- 开发阶段默认导入预生成 Mock；真实实现就位后应通过切换导入路径替换，而不是同时维护多套不一致行为。
- 本 Skill 生成的是"面向 `public/pre_generated_mock/` 与 `public/global_mock/` 的默认内容规范、待落位结果与切换规则"，不是默认拥有 `public/` 写权限的落盘器。
- 若需要把 Mock 正式写入受控目录，只能在授权成立后，由正式目录落位阶段或写文件闸门执行。

## Inputs
- 已验证的接口契约与数据契约。
- 典型请求/响应场景、异常场景、默认数据策略。
- 可选的已有 Mock 稿、切换计划、消费者约束。

## Outputs
- 稳定 Mock 结果：默认面向 `public/pre_generated_mock/` 的内容规范与待落位结果。
- 特殊覆盖边界：何时允许使用 `public/global_mock/` 覆盖默认结果，以及对应的待落位说明。
- Mock 策略说明。
- 替换边界与切换指引。
- 验证入口、阻断项与并行接续入口。

## Action Flow
1. 读取契约，锁定必须被 Mock 覆盖的接口、字段与典型场景。
2. 确定 Mock 策略：覆盖范围、默认数据、异常路径、保真度边界，以及 `pre_generated_mock` 与 `global_mock` 的职责分工；若边界选择存在真实歧义，调用 `AskUserQuestion` 对齐。
3. 生成默认稳定 Mock，并标记哪些行为属于统一默认替身、哪些场景只允许通过 `global_mock` 显式覆盖。
4. 校验 Mock 是否仍与契约一致，是否会误导下游开发，以及切换到真实实现时调用方是否可无痛替换。
5. 输出替换边界、验证入口、覆盖规则、待落位结果与后续切换提示；若需要正式写入 `public/`，显式交给授权落位阶段处理。
6. 调用 `TodoWrite` 回填当前处于"可支撑并行"还是"仍不可判定"。
7. **提醒主线程拉起 GN-004**：若 Mock 已闭合，必须显式提醒主线程：请主线程拉起 GN-004 对本 Mock 产出进行独立审查，确认无硬阻断后方可将 Mock 作为并行开发的稳定依赖。

## hc_sc_ec_mapping
- 主映射：`HC-2` `SC-2` `EC-3`
- 次映射：`EC-1` `SC-3` `EC-6` `EC-7`
- 说明：
  - `HC-2`：Mock 必须是后续并行开发可直接消费的子输入，而不是孤立文件。
  - `SC-2`：其价值在于为多个并行分支提供共同依赖支点。
  - `EC-3`：Mock 是否符合契约必须能被独立验证。
  - `EC-1`：必须定义替换边界与失败回退，不让下游无锚漂移。
  - `SC-3` / `EC-6`：不满足契约时必须显式标成未闭合并解释原因。
  - `EC-7`：Mock 策略、保真度边界或替换逻辑涉及不可自动裁定的取舍时必须请示。

## State Protocol
- `已闭合`：稳定 Mock 已形成，且契约一致性与替换边界说明齐备，可支撑并行开发。
- `未闭合`：Mock 已有初稿，但契约一致性、覆盖范围或替换边界仍不完整，不能稳定下推。
- `当前不可判定`：契约不充分、场景不明确或当前无法确认 Mock 是否能稳定支撑并行。

## Closure Signals
- 存在一组明确可用的 Mock 结果，而不是只有思路说明。
- 已区分 `pre_generated_mock` 默认稳定依赖与 `global_mock` 特殊覆盖边界。
- 明确写出 Mock 策略、替换边界和验证入口。
- 明确给出契约一致性判断。
- 若状态不是 `已闭合`，必须说明当前卡在契约、覆盖还是替换边界。

## hc3_handoff
- 做到哪里：已生成稳定 Mock 与替换边界，或明确说明当前为什么不能产出可用 Mock。
- 为什么做到这里：因为并行开发需要一套稳定替身，而不是各自脑补接口行为。
- 哪些已闭合：已生成的 Mock 结果、已验证的契约一致性、已标注的替换边界。
- 哪些未闭合：仍存疑的 Mock 范围、未定义的切换路径、未对齐的全局 vs 局部 Mock 策略。
- 下一步从哪接：
  - 若 `已闭合`，进入并行开发
  - 若非 `已闭合`，先补契约确认或缩小 Mock 范围

## ec7_trigger_gate
- Mock 是否必要、保真度是否足够、切换边界是否安全无法自动裁定。
- 用户要求直接并行开发，但当前 Mock 尚不足以作为可信替身。
- 契约与 Mock 之间的偏差已经大到会误导开发。

## ec6_minimum_explanation
- 当前为什么可以或不可以将 Mock 作为并行开发入口。
- 已覆盖哪些接口与场景，哪些仍未覆盖。
- Mock 与契约之间的一致性检查结果。
- 建议下一步补什么类型的 Mock，或等待哪些条件满足。

## gn004_touchpoints
- 三件套阶段：检查该 Skill 是否被定义为"并行开发使能器"而非泛测试工具。
- 核心段审查：检查 Mock 策略、契约一致性、替换边界与三值状态是否齐备。
- 执行阶段：检查执行者是否真的以契约一致性作为 Mock 合法性判据，而不是"看起来能跑"。
- 交付前审查：检查正式目录中是否存在稳定 Mock，以及其替换指引是否可执行。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"Mock / 存根 / 预生成 / 测试替身 / 稳定依赖"类触发语义。
- 首步动作明确为"读取契约并确定 Mock 覆盖范围与策略"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0202-generating-stable-mocks`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的触发语义、未命中原因或 `当前不可判定` 理由。
- `首步独立执行`：记录是否能直接从"读取契约并确定 Mock 覆盖范围与策略"启动，以及阻断点。
- `Mock 稳定覆盖`：记录 Mock 结果、`pre_generated_mock` / `global_mock` 分工、替换边界与契约一致性判断是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录引用契约、未覆盖范围、替换风险与后续接续入口。

## fallback_and_escalation
- 若契约未通过，先输出阻断说明，不强行生成 Mock。
- 若 Mock 与契约不一致，先标未闭合，不把偏差交给下游吸收。
- 若替换边界不清，先缩小 Mock 范围或请示，不让 Mock 变成无法卸载的依赖。
- `EC-4` 间接补偿：由 GN-004、契约一致性核对与主 agent 复核防止 Mock 漂移。
- `EC-5` 间接补偿：若当前能力不足以完成 Mock 策略或替换设计，应升级请示，而不是输出不确定的 Mock。
