# s0602 技术债扫描

## 作用
- 对项目中的技术债做分区扫描、证据收集与风险分级。
- 把"哪些债务必须马上治理、哪些只能排入后续队列"说清楚，而不是堆出一份没人执行的清单。
- 为长程实践中的恢复、清理与交付前健康检查提供统一入口。

## Trigger
- 用户要求扫描技术债、清理冗余、排查重复实现、整理遗留问题或做代码体检。
- 当前项目经过多轮迭代，已出现历史草稿、废弃实现、重复工具、注释代码、失配规则或目录漂移风险。
- 交付前、重构前、阶段收束前需要做一次维护视角的健康检查。

## Preconditions
- 当前工作区可读取，且至少能访问目录结构、关键源文件与既有 `.trae/documents/` 留痕。
- 当前任务确实需要做"静态或轻量动态体检"，而不是直接进入大规模删改。
- 已能识别扫描范围或优先级；若项目范围过大且当前没有扫描边界，先标记 `未闭合` 或调用请示做缩圈。
- 若用户要求"边扫边删"，先阻断删除动作，把本 Skill 限定为扫描与分级入口。

## TRAE 特化命令
- 若扫描范围、风险优先级、是否包含历史目录或生成物目录存在真实歧义，必须先调用 `AskUserQuestion` 对齐，再输出扫描结果。
- 若本 Skill 已形成债务分级、治理建议或阻断说明，应调用 `TodoWrite` 同步真实状态，避免主链误判"已经扫描过 = 已经治理完"。
- 本 Skill 只依赖本地目录、代码、规则与历史留痕做检索；优先使用 `Read` / `Grep` / `Glob` / `SearchCodebase`，不把外部搜索当作主要依据。

## Context
- 本 Skill 负责的是"识别与分级"，不是自动删除器、自动重构器或自动归档器。
- 债务扫描必须可回溯到实体证据：文件路径、命中片段、重复位置、未引用入口、规则冲突或历史留痕。
- 结论至少要区分高 / 中 / 低优先级，并说明为什么此刻需要治理或可延后。
- 技术债不仅包括代码冗余，也包括规则漂移、目录残留、测试入口失配、历史草稿失控增长。
- 若某项问题需要修改代码或删除文件，必须在后续另起留痕并通过写文件闸门。本 Skill 是扫描与分级入口，绝对禁止在扫描阶段执行任何删除、修改、重构或归档操作。

## Inputs
- 项目目录结构与关键代码文件。
- 既有 `.trae/documents/` 历史记录、过往治理记录与相关规则文件。
- 可选的扫描边界、时间窗口、重点模块或高风险目录。
- 可选的历史报告、重复文件线索或交付前检查清单。

## Outputs
- 技术债扫描报告。
- 债务项清单：含类型、证据、影响范围、优先级与建议动作。
- 高风险阻断项与可延期项区分结果。
- 后续治理入口：需要新建留痕、需要提醒主线程拉起 GN-004 抽查、需要推迟到后续批次的说明。
- 三值状态、未闭合项与接续入口。

## Action Flow
1. 读取扫描范围、目录结构、既有留痕与相关规则，确认本次体检边界。
2. 若扫描范围、风险优先级、是否包含历史目录或生成物目录存在真实歧义，必须先调用 `AskUserQuestion` 对齐，不得在未获人类裁决前开始扫描。
3. 分批扫描冗余文件、备份/临时文件、重复实现、注释尸体、未引用入口、规则失配与测试入口失效等候选债务。
4. 为每类命中项收集实体证据，并按高 / 中 / 低优先级做分级，明确"立即治理 / 后续排期 / 仅观察"。
5. 若某类债务是否影响当前交付判断、是否必须立即治理存在真实歧义且当前无法自动裁定，必须先调用 `AskUserQuestion` 对齐，不得自行判定。
6. 输出结构化债务清单、证据路径、优先级、建议动作与需要单独留痕的后续治理入口。
7. 若用户要求直接删除、重构或大批量清理，必须先阻断删除/清理动作，并向人类请示治理策略。本 Skill 限定为扫描与分级入口，不得在扫描阶段执行任何清理操作。
8. 调用 `TodoWrite` 回填当前处于"扫描已闭合"还是"仍待补范围/补证据"。

## hc_sc_ec_mapping
- 主映射：`HC-1` `EC-3` `EC-2`
- 次映射：`HC-3` `SC-3` `EC-6` `EC-7`
- 说明：
  - `HC-1`：先回答当前是否形成足够可靠的债务识别结果，而不是把"扫了一遍"冒充闭合。
  - `EC-3`：每个债务项都必须有实体证据可复核，不能只靠主观印象。
  - `EC-2`：通过优先级分级限制治理范围，避免一轮体检演化成失控清理。
  - `HC-3`：交代本轮已经扫到哪里、哪些目录或类型仍未覆盖。
  - `SC-3`：证据不足时只能标记未闭合或当前不可判定，不能假装问题已坐实。
  - `EC-6/EC-7`：扫描范围、处置优先级或交付影响无法自动裁定时必须解释并请示。

## State Protocol
- `已闭合`：扫描范围内的主要债务类型已检查并形成分级结果，证据与后续治理入口齐备。
- `未闭合`：扫描已开始，但范围、证据、分级或后续治理入口仍不完整，不能宣告体检完成。
- `当前不可判定`：当前无法确认扫描范围、关键目录不可读或证据不足以形成可靠债务判断。

## Closure Signals
- 存在结构化债务清单，而不是只有零散观察。
- 每个高优先级债务项至少包含类型、证据、影响范围、优先级与建议动作。
- 明确区分"立即治理 / 后续排期 / 仅观察"。
- 明确给出需要另起留痕的后续治理入口，而不是直接在扫描阶段动刀。
- 若状态不是 `已闭合`，必须说明当前卡在范围、证据、优先级还是治理接续设计。

## hc3_handoff
- 做到哪里：已完成本轮扫描与分级，或明确说明当前为什么无法可靠完成体检。
- 为什么做到这里：因为长程实践中的项目衰退往往来自积累性债务，而不是单一 bug。
- 哪些已闭合：已扫描的目录、已收集的证据、已分级的债务项与已确认的治理入口。
- 哪些未闭合：尚未覆盖的目录、证据不足的候选项、仍待人工裁定的优先级与清理边界。
- 下一步从哪接：
  - 若 `已闭合`，进入后续专项治理留痕、交付前处理或排期管理
  - 若非 `已闭合`，先缩小范围、补证据或明确扫描边界后再继续

## ec7_trigger_gate
- 无法确认本轮扫描应覆盖哪些目录、历史草稿或生成物。
- 某类债务是否会影响当前交付判断，当前无法自动裁定。
- 用户要求直接删除、重构或大批量清理，但当前仅完成扫描未完成治理留痕。

## ec6_minimum_explanation
- 当前为什么可以或不可以把本轮体检视为已完成。
- 已扫描哪些类型和目录，哪些仍未覆盖。
- 已收集哪些关键证据，哪些项仍只是候选观察。
- 建议下一步优先治理什么、哪些应延后、哪些需要另起留痕或审查。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否被定位为"维护体检入口"，而不是自动重构工具。
- 执行阶段：检查执行者是否真的基于实体证据做分级，而不是把模糊印象写成债务结论。
- 关键检查点阶段：当高优先级债务已影响执行质量、交付判断或恢复能力时，应提醒主线程拉起 GN-004 抽查。
- 交付前审查：检查高优先级债务是否已被显式处理、挂起或留痕，而不是被静默忽略。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"技术债 / 冗余检查 / 代码体检 / 重复实现 / 遗留清理"类触发语义。
- 首步动作明确为"读取扫描范围并识别候选债务类型与证据来源"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0602-scanning-technical-debt`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的技术债 / 代码体检语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"读取扫描范围并识别候选债务类型与证据来源"启动，以及阻断点。
- `债务扫描覆盖`：记录冗余文件、重复实现、注释尸体、未引用入口、规则失配、优先级分级与治理入口是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录证据路径、观察项、未覆盖范围、后续留痕入口与治理优先级说明。

## fallback_and_escalation
- 若扫描范围过大，先分批次扫描高风险目录与高价值模块，不搞一次性全仓粗暴体检。
- 若某项候选问题缺证据，只保留为观察项或未闭合项，不把猜测写成结论。
- 若发现高优先级债务需要改代码或删文件，必须转入新的留痕与写文件闸门，不在本 Skill 内直接处理。
- `EC-4` 间接补偿：由 GN-004、规则核对与主 agent 复核防止执行者把"看起来乱"写成未经证实的债务结论。
- `EC-5` 间接补偿：若当前能力不足以判断复杂重复实现、交付影响或清理边界，应升级请示，而不是输出看似完整的治理结论。
