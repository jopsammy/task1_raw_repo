# s0103 方案融合定稿

## 作用
- 把多方案比较结果收束为一个可继续推进的最终方案。
- 显式写出保留什么、放弃什么、为什么这么取舍。
- 分歧过大或存在本质冲突时硬阻断，不强行定稿。
- 把结果整理为 `s0201` 可直接使用的定稿输入，而不是停在评语层。

## Trigger
- `s0102` 已生成多方案。
- 用户要求方案融合、方案选择、输出最终方案。
- 当前任务必须从探索阶段进入可执行的收束阶段。

## Preconditions
- 当前至少已经存在两条以上可比较的候选方案，以及对应的差异、风险或适用场景说明。
- 下游节点确实要求单一方向输入，例如契约生成、模块拆分或执行编排；若仍处于开放探索期，不启动本 Skill。
- 当前取舍所需的关键约束、风险边界、不可妥协项至少已有一版显式表达；若没有，先补约束或请示。
- 若候选方案质量过低、差异不清或关键价值裁决完全缺席，直接标记 `未闭合` / `当前不可判定`，不得伪造定稿。
- **融合可行性最低条件**：候选方案之间不存在本质性互斥——若两个方案在核心架构上互相排斥且无法消解，不满足融合前提，必须硬阻断。

## TRAE 特化命令
- 当多个候选方案之间的取舍明显涉及用户偏好、风险容忍度或成本边界时，必须先调用 `AskUserQuestion`，再完成定稿；不能把价值裁决偷偷内嵌为默认选择。
- 当本 Skill 遇到硬阻断（分歧过大/本质冲突/互斥不可消解），必须在向人类汇报阻断原因、冲突清单与回退路径后，再调用 `TodoWrite` 标记状态，不得绕过阻断继续执行。
- 当本 Skill 已输出可继续进入下游的单一方案或明确的阻断说明时，应调用 `TodoWrite` 同步当前状态，避免执行链误把"探索已收束"与"仍在比较中"混为一谈。
- 本 Skill 的信息归并与比对必须基于本地实体和当前方案文本完成；不得把"定稿裁决"外包给非本地检索代理。

## Context
- 本 Skill 可以组织取舍，但不替代人类的最终价值裁决。
- 输出必须包括结果和理由，不能只有结论。
- 若关键冲突仍未解决，必须保留未闭合项，不得伪造定稿。
- 结果必须可直接成为后续契约与模块拆分的输入。
- 硬阻断不是失败——它是合法的终止态，说明当前多方案阶段不具备融合条件。硬阻断时必须输出完整的阻断原因与冲突清单，让人类知道"卡在哪、为什么卡、回去补什么"。

## Inputs
- 多个候选方案。
- 差异表、风险表、适用场景说明。
- 可选的用户偏好、资源上限、不可妥协约束。

## Outputs
- 最终方案定稿（若融合成功）。
- 取舍理由与融合点列表。
- 未闭合项、残余风险、下一步接续入口。
- 面向 `s0201` 的结构化下游输入。
- 若硬阻断，则输出阻断原因、冲突清单、建议回退路径。

## Action Flow
1. **共识分析**：读取所有候选方案和差异表。对每个决策维度做共识判定——显式区分 `共识`（各方案一致）、`模糊共识`（倾向一致但细节不同）、`非共识`（方向相反或结构不同）。
2. **硬阻断判定**：在共识分析完成后，先判断是否触发硬阻断条件。以下任一成立则必须硬阻断：
   - **分歧过大**：在核心架构、数据模型或关键约束维度上，至少两个方案存在方向性相反且不可调和的分歧。
   - **本质性冲突**：方案之间存在明确的互斥关系，选择其一意味着另一方案的核心理念完全无法保留。
   - **互斥不可消解**：尝试过互补/折中分析但仍无可行融合路径。
   - **候选方案质量不足**：差异不清、覆盖度严重不足、或方案本身不完整到无法做有意义的融合。

   硬阻断时：向人类明确汇报阻断原因、冲突清单（逐条列出哪个方案和哪个方案在哪一维度冲突）、建议回退路径（通常指向 `s0102` 重新分叉或补约束）。调用 `TodoWrite` 标记 `未闭合`。**不得在硬阻断状态下继续执行后续步骤。**

3. **互补/互斥分析**：若未触发硬阻断，则分析方案之间哪些元素是互补的（可合并）、哪些是互斥的（需选择一边）。列出可融合点与必须裁决的互斥点。
4. **人类裁决**：对于互斥点或有真实价值取舍的融合选择，调用 `AskUserQuestion` 向人类提供结构化选项（2-4 个取舍方案）。不得自行裁决互斥点或把个人偏好伪装成默认融合路径。
5. **融合**：基于人类裁决结果融合各方案的可互补元素，形成单一最终方案。保留取舍理由。
6. **输出**：输出最终方案定稿（含保留项、舍弃项及其理由）、未闭合项、残余风险、下游接续入口。
7. **提醒主线程拉起 GN-004**：融合完成后，必须显式提醒主线程：请主线程拉起 GN-004 对本融合定稿进行独立审查。审查内容至少含：
   - 融合策略是否偏离用户原始意图
   - 是否真实多方案融合（非单方案伪装）
   - 融合策略是否合理
   - 最终方案是否对齐用户意图
   - 直至 GN-004 确认无硬阻断后才可宣告闭合。绝对禁止半成品汇报。
8. 调用 `TodoWrite` 回填"已收束 / 未收束 / 当前不可判定"。

## hc_sc_ec_mapping
- 主映射：`HC-1` `SC-1` `EC-6`
- 次映射：`HC-3` `EC-7`
- 说明：
  - `HC-1`：必须回答当前是否已经有足够理由定稿或必须硬阻断。
  - `SC-1`：定稿不是形式闭合，而是价值与约束层面的收束；硬阻断也不是失败，而是收束态的一种。
  - `EC-6`：必须解释为什么保留这些、放弃那些，或为什么必须硬阻断。
  - `EC-7`：遇到关键取舍无法自动裁决时，或硬阻断时需要人类确认回退路径时，必须请示。

## State Protocol
- `已闭合`：已形成单一可继续推进的最终方案，并附有清晰取舍理由，且已通过 GN-004 审查（无硬阻断）。
- `未闭合`：已有收束方向但仍有关键冲突或价值取舍未决，或已硬阻断但等待人类确认回退路径。
- `当前不可判定`：现有候选方案不足以支持可信定稿，或关键输入缺失。

## Closure Signals
- 存在一份单一最终方案，而不是继续并列多个候选。
- 明确列出保留项、舍弃项和理由。
- 明确指出残余风险与未闭合项。
- 输出结构足以直接进入 `s0201`。
- 已显式提醒主线程拉起 GN-004 审查，且 GN-004 返回无硬阻断。
- 若为硬阻断闭合：已输出完整的阻断原因、冲突清单与建议回退路径。

## hc3_handoff
- 做到哪里：已将探索阶段收束成最终方案，或说明为什么当前无法定稿（硬阻断）。
- 为什么做到这里：因为下游契约与模块拆分必须建立在单一可执行方案上。
- 哪些已闭合：已确认的架构路线、确定保留的核心要素、主要取舍结论。
- 哪些未闭合：仍需人类裁定的价值取舍、残余冲突、资源不确定项。
- 下一步从哪接：
  - 若 `已闭合`，提醒主线程 GN-004 审查后进入 `s0201`
  - 若硬阻断，回到 `s0102` 重新分叉或补约束
  - 若非 `已闭合`，先补裁决或回退到 `s0102`

## ec7_trigger_gate
- 多方案之间存在关键冲突，无法自动完成取舍。
- 价值判断、成本边界、风险容忍度明显需要人类裁定。
- 用户要求立即定稿，但当前信息不足以支持可信收束。
- 硬阻断状态需要人类确认回退路径。
- 关键缺口存在但未拉起 AskUserQuestion → 必须阻断。

## ec6_minimum_explanation
- 当前为什么可以或不可以定稿（或必须硬阻断）。
- 保留了哪些方案元素，舍弃了哪些，理由是什么。
- 若硬阻断：逐条列出冲突维度、冲突方案、为何无法消解。
- 还有哪些风险与未闭合项。
- 建议下一步进入哪一个下游节点或回退到哪个上游节点。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否把"定稿与理由"和"硬阻断与冲突清单"写成刚性输出。
- 核心段审查：检查是否存在假融合、假闭合或只给结论不给理由；检查硬阻断条件是否完整且可执行。
- 执行阶段：检查执行者是否真的完成共识分析、硬阻断判定、取舍暴露、理由展开与未闭合项保留，而不是把个人偏好伪装成默认定稿。
- 交付前审查：检查其结果是否可直接服务 `s0201`。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"方案融合 / 最终方案 / 方案选择 / 收束定稿 / 硬阻断 / 共识分析"等触发语义。
- 首步动作明确为"读取差异表并做共识分析+硬阻断判定"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0103-merging-schemes`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的方案融合 / 收束定稿 / 硬阻断 / 共识分析语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"读取差异表并做共识分析+硬阻断判定"启动，以及阻断点。
- `融合定稿核对`：记录共识/模糊共识/非共识分类、硬阻断触发情况、取舍理由、保留项、舍弃项、未闭合项与下游接续入口是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录候选方案、冲突项、未决项、硬阻断原因与后续接续入口。

## fallback_and_escalation
- 若关键冲突无法消解，先输出冲突与未决项，不强行定稿。
- 若触发硬阻断（分歧过大/本质冲突/互斥不可消解），**必须硬阻断并向人类汇报**阻断原因、冲突清单与建议回退路径。回退路径指向 `s0102`——让 s0102 基于新的约束或分叉维度重新生成方案。
- 若当前候选方案质量不足，回退到 `s0102` 重新分叉。
- `EC-4` 间接补偿：由 GN-004、主 agent 与取舍理由检查防止"结论先行"。
- `EC-5` 间接补偿：遇到无法自动裁决的价值取舍时升级请示，而不是自作主张闭合。
