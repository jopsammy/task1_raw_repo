# s0101 需求闭合结构化

## 作用
- 把原始需求转成后续 `s0102` 可直接消费的结构化输入。
- 把"还能不能继续往下做"作为主问题，而不是只做格式整理。
- 在需求仍不充分时，提供明确的阻断说明、澄清清单与接续入口。

## Trigger
- 用户给出原始需求，但需求仍是自然语言散点。
- 用户要求先做需求结构化、需求闭合检查、进入方案设计前置整理。
- 当前任务已经出现"可以写点东西，但还不能稳定进入方案设计"的状态。

## Preconditions
- 当前输入中至少存在一个可识别的任务目标或问题域，而不是完全空白的占位指令。
- 至少已经拿到原始需求文本、会议纪要、历史稿件中的一种，可作为需求收束起点。
- 当前任务的下游确实需要结构化需求输入；若只是闲聊、纯灵感记录或无需进入方案设计，不启动本 Skill。
- 若核心上下文完全缺失到连"要收束什么"都无法判断，直接进入 `当前不可判定` 或先触发 `AskUserQuestion`，不伪造启动条件。

## TRAE 特化命令
- 若需求存在关键歧义、边界冲突、成功标准缺失，必须先调用 `AskUserQuestion` 做结构化澄清，不能靠自然语言闲聊补完。
- 若当前任务已进入执行链路，且本 Skill 已形成可交付结果或阻断说明，应调用 `TodoWrite` 同步"已闭合 / 未闭合 / 当前不可判定"的真实状态。
- 本 Skill 的本地检索只能使用 `Read` / `Grep` / `Glob` / `SearchCodebase`；不得把需求整理任务外包给非本地检索代理。

## Context
- 本 Skill 不替代人类确认需求，也不代替最终价值判断。
- 输出必须兼容后续方案生成，不得只给散文式总结。
- 输出必须显式携带三值状态、未闭合项与下一步接续点。
- 若关键输入缺失，允许阻断，不允许伪造完整需求。

## Inputs
- 用户原始需求文本。
- 已知约束、目标、边界、成功标准。
- 可选的历史需求稿、会议纪要、任务上下文。

## Outputs
- 结构化需求摘要。
- 需求闭合判断。
- 澄清清单与阻断项。
- 供 `s0102` 使用的候选输入骨架。

## Action Flow
1. 提取任务目标、约束条件、成功标准、已知边界。
2. 分离"已知事实""推断项""未知但关键项"。
3. 将需求整理为可继续推进的结构化块。
4. 若关键缺口足以阻断下游，必须先调用 `AskUserQuestion` 对齐，不得自行补完缺口或静默保持阻断状态向人类汇报"已完成"。
5. 输出当前三值状态，并明确哪些项阻止进入下游方案设计。
6. 给出下一步接续建议：进入 `s0102`，或先请示 / 澄清。

## hc_sc_ec_mapping
- 主映射：`HC-1` `HC-2` `HC-3`
- 次映射：`SC-3` `EC-6` `EC-7`
- 说明：
  - `HC-1`：先回答"当前需求是否已足够支持继续推进"。
  - `HC-2`：把需求拆成后续可用的检查点输入。
  - `HC-3`：交代当前做到哪里、为什么停在这里、下一步从哪接。
  - `SC-3`：禁止把缺口伪装成已闭合。
  - `EC-6/EC-7`：关键歧义必须解释并请示。

## State Protocol
- `已闭合`：目标、约束、成功标准、关键未决项已被显式整理，且可以稳定进入 `s0102`。
- `未闭合`：已识别出明确缺口，但仍需用户或上游补充后才能推进。
- `当前不可判定`：输入过少、上下文冲突或边界不成立，当前无法确认是否可继续。

## Closure Signals
- 存在一份结构化需求结果，至少覆盖：目标、约束、成功标准、未决项、下一步。
- 明确给出三值状态，而不是只有"整理完成"。
- 若状态不是 `已闭合`，必须写出阻断原因和后续动作。

## hc3_handoff
- 做到哪里：需求已被压缩为结构化输入骨架。
- 为什么做到这里：因为继续方案设计前，必须先确认需求缺口是否已被显式暴露。
- 哪些已闭合：已知目标、已知约束、已知成功标准。
- 哪些未闭合：仍缺口的业务规则、资源条件、优先级裁决。
- 下一步从哪接：
  - 若 `已闭合`，进入 `s0102`
  - 若 `未闭合` 或 `当前不可判定`，先补信息或触发请示

## ec7_trigger_gate
- 需求中存在影响方案方向的关键歧义。
- 约束之间互相冲突，无法安全整理为单一输入。
- 关键缺口存在但未拉起 `AskUserQuestion` 则必须阻断——静默保持阻断状态且不向人类请示，视同违规。
- 用户要求继续推进，但当前输入不足以支持真实闭合。
- EC-7 闭合条件：所有阻塞性缺口均已通过 `AskUserQuestion` 获得人类裁决后才可退出 EC-7。

## ec6_minimum_explanation
- 当前为什么不能直接进入下游。
- 哪些事实已知，哪些仍未知。
- 已尝试做了哪些整理。
- 建议用户或上游下一步补什么。

## gn004_touchpoints
- 三件套阶段：检查该 Skill 是否把"需求闭合"而非"格式整理"作为主目标。
- 核心段审查：检查三值状态、闭合信号、HC-3 语义标注是否齐备。
- 执行阶段：检查执行者是否真的先做需求收束与缺口暴露，而不是跳过本 Skill 直接把伪需求送入 `s0102`。
- 交付前审查：检查其输出是否真的能作为 `s0102` 的输入。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"需求结构化 / 需求闭合 / 澄清前置"类触发语义。
- 首步动作明确为"提取目标、约束、成功标准与缺口"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0101-requirement-closure-structuring`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的需求结构化 / 需求闭合 / 澄清前置语义，或未命中原因。
- `首步独立执行`：记录是否能直接从"提取目标、约束、成功标准与缺口"启动，以及阻断点。
- `需求闭合核对`：记录目标、约束、成功标准、缺口清单、三值状态与下游接续入口是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录引用需求、缺口项、冲突项与后续接续入口。

## fallback_and_escalation
- 若需求过于模糊，先产出澄清清单，不强行补完。
- 若约束冲突，触发请示，不继续向 `s0102` 传递伪输入。
- `EC-4` 间接补偿：由 GN-004 审查、AskUserQuestion 与主 agent 复核防止偏航。
- `EC-5` 间接补偿：遇到能力边界时升级给用户或上位审查，而不是自行裁定。
