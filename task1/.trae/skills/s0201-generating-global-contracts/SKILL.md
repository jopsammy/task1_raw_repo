# s0201 全局契约生成

## 作用
- 把最终方案固化为数据契约、接口契约、配置契约三层唯一真相源。
- 为 `s0202`、`s0203` 以及后续 GN-004 审查提供共同依赖锚点。
- 把"契约是否一致且可继续推进"作为核心判断，而不是停留在文档补写层。

## Trigger
- 最终方案已经形成，需要生成统一契约作为下游执行锚点。
- 用户要求定义接口、Schema、配置边界或契约骨架。
- 当前任务若继续推进 Mock、模块拆分或执行安排，会因缺少统一契约而产生理解漂移。

## Preconditions
- 上游已经存在一版明确的最终方案输入，例如 `s0103` 的定稿结果或等价的冻结架构说明。
- 至少可以识别核心实体、关键接口边界、主要配置项三类信息中的大部分；若缺口过大，先补方案而不是硬写契约。
- 当前确实需要把"方案语义"转为"可验证的契约边界"；若仍处于开放探索期，不启动本 Skill。
- 若方案尚未冻结或关键边界仍在频繁变化，直接输出阻断说明或 `当前不可判定`，不伪造正式契约。
- 若当前已存在旧版与新版契约差异，需要做的是变更影响面识别、同步顺序和阻断判断，则优先转入 `s0601`，不把"变更适配"伪装成"首轮契约冻结"。

## TRAE 特化命令
- 若发现最终方案是否冻结、接口边界归属、关键配置责任仍存在真实歧义，必须先调用 `AskUserQuestion` 对齐，再落正式契约。
- 若本 Skill 已产出可供下游消费的契约集或明确阻断说明，应调用 `TodoWrite` 同步"已闭合 / 未闭合 / 当前不可判定"的真实状态。
- 本 Skill 的契约抽取与核对只能基于本地方案文本、现有实体文件与当前任务上下文完成；检索仅使用 `Read` / `Grep` / `Glob` / `SearchCodebase`。

## Context
- 三层契约必须互相一致，不能各写各的。
- 本 Skill 不负责替代真实实现，但必须为实现与验证定义唯一边界。
- 输出必须能直接服务 Mock、模块拆分与后续独立审查。
- 一旦发现契约冲突，应优先阻断，不允许用"后续再改"掩盖假闭合。
- 数据契约必须落为 `JSON Schema`，接口契约必须落为 `.pyi` 存根，配置契约也必须用 `JSON Schema` 表达默认值与补齐规则。
- 三层契约必须显式覆盖错误码与异常契约，不能只定义"正常路径"。
- 配置默认值与缺失字段自动补齐必须在契约层先说明，禁止把关键配置硬编码到业务实现。
- 本 Skill 生成的是"面向 `public/` 的正式契约内容规范与待落位实体"，不是默认拥有 `public/` 写权限的落盘器。
- 若需要把契约真正写入正式目录或受控 `public/` 区域，只能在授权成立后，由正式目录落位阶段或写文件闸门执行。

## Inputs
- 最终方案定稿或等价的冻结架构说明。
- 核心实体、接口说明、配置边界、非功能约束。
- 可选的历史契约稿、旧版 schema、约束清单。

## Outputs
- 数据契约：面向 `public/schema/` 的 `JSON Schema` 内容规范与待落位实体。
- 接口契约：面向 `public/interface_stub/` 的 `.pyi` 存根内容规范与待落位实体。
- 配置契约：面向 `public/config_template/` 的配置 `JSON Schema` 内容规范与待落位实体。
- 错误码与异常契约。
- 契约一致性判断、未闭合项与下游接续入口。

## Action Flow
1. 读取最终方案，锁定不可违背的实体边界、接口职责与配置责任。
2. 提取三层契约骨架：数据 `JSON Schema`、接口 `.pyi` 存根、配置 `JSON Schema`，同时列出错误码与异常契约。
3. 为数据与配置字段补齐类型、范围、必填性、默认值与自动补齐规则；为接口定义参数、返回值与抛出异常。
4. 校验三层之间是否存在命名冲突、字段冲突、边界冲突、职责重叠或异常语义冲突。
5. 若关键冲突或冻结条件不成立，调用 `AskUserQuestion` 对齐，或明确保持阻断状态。
6. 输出三层契约内容规范、待落位实体及其一致性结论，并标明哪些下游节点可以继续消费；若需要正式写入 `public/`，显式交给授权落位阶段处理。
7. 调用 `TodoWrite` 回填当前处于"可进入 Mock / 模块拆分"还是"仍不可判定"。
8. **提醒主线程拉起 GN-004**：若契约已闭合，必须显式提醒主线程：请主线程拉起 GN-004 对本契约产出进行独立审查，确认无硬阻断后，方可将本契约作为下游唯一边界。在 GN-004 审查通过前，不得将契约下推至 s0202/s0203。

## hc_sc_ec_mapping
- 主映射：`HC-2` `HC-3` `EC-3`
- 次映射：`SC-3` `EC-6` `EC-7`
- 说明：
  - `HC-2`：把最终方案分解成后续多个节点都可复用的检查点输入。
  - `HC-3`：契约不仅要给结果，还要交代当前冻结到了哪里、哪里仍未闭合。
  - `EC-3`：契约作为唯一真相源，必须接受独立一致性审查。
  - `SC-3`：契约不一致或缺边界时不得伪装成已完成。
  - `EC-6/EC-7`：关键冲突必须解释并请示，不得静默吞掉。

## State Protocol
- `已闭合`：三层契约均已形成，且一致性检查通过，可稳定服务下游。
- `未闭合`：契约骨架已形成，但仍有冲突、缺口或冻结条件未满足，不能直接下推。
- `当前不可判定`：当前输入不足以可靠生成三层契约，或是否冻结仍无法确认。

## Closure Signals
- 存在一套覆盖数据、接口、配置三层的契约结果。
- 数据契约为 `JSON Schema`，接口契约为 `.pyi`，配置契约含默认值与自动补齐规则。
- 错误码与异常契约已被显式纳入，而不是遗漏到实现阶段。
- 明确给出一致性结论，而不是只列若干片段。
- 明确写出未闭合项、冲突项和下游接续入口。
- 若状态不是 `已闭合`，必须说明为什么当前不能作为唯一真相源。

## hc3_handoff
- 做到哪里：已把最终方案压缩为三层契约，或明确说明当前为什么无法完成压缩。
- 为什么做到这里：因为后续 Mock、模块拆分与执行安排必须建立在统一边界上，而不是各自解释方案。
- 哪些已闭合：已冻结的实体定义、接口职责、配置边界与一致性检查结论。
- 哪些未闭合：仍冲突的字段、未裁定的接口责任、尚未冻结的配置约束。
- 下一步从哪接：
  - 若 `已闭合`，提醒主线程 GN-004 审查后进入 `s0202` 与 `s0203`
  - 若非 `已闭合`，先补裁决或回退到上游定稿节点

## ec7_trigger_gate
- 是否已冻结架构无法确认。
- 三层契约之间出现关键冲突，无法自动统一。
- 用户要求继续推进下游，但当前契约还不能作为唯一真相源。

## ec6_minimum_explanation
- 当前为什么可以或不可以把方案落成正式契约。
- 哪些边界已经被固定，哪些仍存在冲突或空洞。
- 已完成了哪些一致性检查。
- 建议下一步由谁补齐什么，才能继续进入下游。

## gn004_touchpoints
- 三件套阶段：检查本 Skill 是否被定位为"唯一真相源生成器"而非普通文档整理器。
- 核心段审查：检查三层契约、一致性结论、三值状态与 HC-3 标注是否齐备。
- 执行阶段：检查执行者是否真的以契约作为下游唯一边界，而不是在 Mock 或模块拆分时各自重解释方案。
- 交付前审查：检查契约结果是否仍能回溯到最终方案，并支撑正式目录中的下游 Skill。
- 所有 gn004_touchpoints 段落中的审查动作均为主线程拉起 GN-004，不得暗示 subagent 可自行拉取。

## installability_validation
- `frontmatter` 存在且字段可读。
- `description` 能命中"契约生成 / schema / 接口定义 / 唯一真相源"类触发语义。
- 首步动作明确为"锁定最终方案边界并抽取三层契约骨架"，主执行者可独立执行。
- 若当前环境无法验证触发命中，则状态记为 `当前不可判定`。

## validation_record_format
- `验证对象`：`s0201-generating-global-contracts`
- `验证时间`：记录实际执行时间戳。
- `frontmatter 合规`：`通过 / 失败 / 当前不可判定`
- `description 触发命中`：记录命中的触发语义、未命中原因或 `当前不可判定` 理由。
- `首步独立执行`：记录是否能直接从"锁定最终方案边界并抽取三层契约骨架"启动，以及阻断点。
- `三层契约覆盖`：记录 `JSON Schema`、`.pyi`、配置 `JSON Schema`、错误码与异常契约是否齐备。
- `总体结论`：只能填写 `已闭合 / 未闭合 / 当前不可判定` 之一。
- `证据与备注`：记录引用文件、冲突项、未闭合项与后续接续入口。

## fallback_and_escalation
- 若最终方案未冻结，先输出阻断说明，不强行生成正式契约。
- 若三层契约冲突，先保留冲突与请示入口，不把伪一致性交给下游。
- 若 `JSON Schema`、`.pyi`、配置默认值或异常契约缺任一关键层，直接保持 `未闭合`，不以"先写主体"冒充契约冻结。
- 若当前需要的是"旧新契约差异适配"而非"首轮契约冻结"，直接回退到 `s0601`，不混用两个 Skill 的职责。
- 若缺少对 `public/` 的正式写权限，仅输出待落位实体与落位前置条件，不越权直接落盘。
- `EC-4` 间接补偿：由 GN-004、一致性检查与主 agent 复核防止契约偏航。
- `EC-5` 间接补偿：若当前能力不足以完成边界裁决或一致性统一，应升级请示，而不是自造唯一真相源。
