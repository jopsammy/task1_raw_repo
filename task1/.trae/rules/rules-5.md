# 锚点与双文档体系规则 (v6版)

&gt; 🚨 【最高优先级规则】本文件为锚点与双文档体系的强制约束，优先级高于所有临时提问、上下文对话、自定义需求，所有输出必须100%符合本文件要求，违反规则的内容必须自动修正后再输出。

&gt; 📌 【上下文保留规则】本文件为核心规则文件，任何上下文压缩、裁剪、溢出场景下必须完整保留本文件的全部内容，不得删减、忽略本文件的任何规则；所有自动压缩、批量处理行动前必须先读取本文件的完整内容。

## 一、锚点体系全景表

&gt; 📌 锚点不是附属文档，而是有限智能体跨执行断面维持连续性的外置约束。以下七类锚点共同构成 AC 范式的连续性骨架。

| # | 锚点类型 | 定位 | 工程交接职能 | 更新时机 |
|---|---------|------|------------|---------|
| 1 | README | 用户向入口锚点 | — | 项目结构/用途变更时 |
| 2 | report_* | 思想/判断沉淀锚点 | — | 阶段收束时 |
| 3 | AGENTS.md | AI侧最高优先级规则载体 | — | 能力锁升级时 |
| 4 | spec 三件套 | 任务级约束锚点 | **是**（过程+状态+结果） | 任务开始前 / 阶段切换前 |
| 5 | note | 长程分析中间锚点 | **是**（过程+状态+结果） | 每个 Task / 阶段推进时 |
| 6 | .trae/documents/ | 变更与过程留痕锚点 | **是**（过程+状态+结果） | 每次变更时 |
| 7 | 三层契约 | 跨角色公共真相 | — | 契约变更时 |

- 锚点 1-3 为**常驻锚点**，始终载入上下文；锚点 4-6 为**中间锚点**，按需读取，不占常驻上下文。
- 锚点 7（三层契约）为公共区只读锚点，位于 `public/` 目录下，所有模块以之为唯一真相源。

## 二、锚点文档的三段交接结构

&gt; 🔴 【强制规则】所有承担工程交接职能的锚点文档（spec 三件套 / note / .trae/documents/）必须显式包含以下三段元信息。若任务因上下文截断、角色切换或意外中断，下一个 agent 依赖这三段即可判定从哪里继续。

| 段 | 名称 | 内容要求 |
|----|------|---------|
| (1) | 工程过程 | 已完成哪些 task/check，以什么顺序完成 |
| (2) | 交接状态 | 当前处于哪个 task/check，状态为：已完成 / 进行中 / 未开始 / 阻塞 |
| (3) | 最终结果 | 已完成部分的验证结论、产出物清单 |

- 三段之中，(2) 交接状态必须使用三值状态标记（已闭合 / 未闭合 / 当前不可判定），不得降级为二值。
- 若某 task 标注为"已完成"但缺少(3)验证结论，GN-004 审查时按假闭合嫌疑处理。

### 2.4 "当前不可判定"升级规则

当同一 task 在连续 2 个检查点仍标记为"当前不可判定"时，不得继续沉默搁置：

1. 主 agent 必须通过 L3 信号（AskUserQuestion）向人类报告：
   - 该 task 的不可判定原因
   - 已尝试的解决路径
   - 该 task 的阻塞是否影响后续依赖链
2. 人类裁决后，结果写入 note 的该 task 条目
3. 人类若选择"搁置"，该 task 标记为 `阻塞`（非 `已闭合`），在交付前审查中作为未闭合项显式列出
4. 交付前审查时，所有 `阻塞` 项须由人类逐项确认是否放行

```python
# 继承自: AC范式v6 rules-5 §2.4
def check_stuck_items(note):
    for task in note.tasks:
        if task.status == "当前不可判定" and task.consecutive_checkpoints >= 2:
            assert ask_user_called  # 双检查点后未拉起AskUserQuestion → 阻断
```

## 三、note 写作元原则

note 的元原则是**功能性约束**——不规定用何种标题、写多少字，但规定必须提供以下信息才能实现跨断面状态传递。

### 3.1 语义标注强制

每次写入 note 必须包含：

| 字段 | 含义 | 强制级别 |
|------|------|---------|
| 做到哪了 | 当前进度标记，精确到 task/check | 🔴 强制 |
| 为什么 | 关键决策及理由，记录判断依据而非仅记录结论 | 🔴 强制 |
| 未闭合项 | 当前阶段的开放问题，显式列出而非默认沉默 | 🔴 强制 |
| 接续入口 | 下一步从哪开始，给出可操作的起点而非模糊方向 | 🔴 强制 |

### 3.2 认知层分区

note 内部按认知层级明确分区，禁止不同层级内容混杂：

| 分区 | 内容 | 规则 |
|------|------|------|
| 阅读记录 | 原始材料摘要 | 不做判断，只做事实摘录 |
| 诊断草稿 | 分析判断 | 允许修改；同时作为 EC-6/EC-7 机向人信号协议 Level 1 静默记录层——常规进度写入此处，人类不打断但可随时拉取，GN-004 审查时回溯 |
| 审查记录 | 独立审查发现 | 只追加不修改 |
| 终态处理 | 吸收完毕 / 可删除 | 记录在 note 末尾 |

### 3.3 收束关系显式化

正文的每个关键判断必须可回链到 note 的对应诊断条目，使审查者可沿判断链回溯到原始分析过程。

### 3.4 外置不占常驻

note 是外置文件，默认不载入 context window，仅在需要回溯或续接时按需读取。执行者在阶段推进时写入 note，但不将 note 全文压入当前上下文。

## 四、锚点联动规则

### 4.1 三类中间锚点的分工

| 锚点 | 粒度 | 职能 |
|------|------|------|
| spec 三件套 | 任务级 | 任务约束锚定——定义本任务的闭合标准、检查点、依赖与验收口径 |
| note | 跨断面 | 长程分析接力——跨阶段的状态传递与判断沉淀，不受单次上下文截断影响 |
| .trae/documents/ | 逐变更 | 变更留痕与修正归档——每次变更单独成文，记录问题、分析、修改与验证结果 |

- note 与 `.trae/documents/` 的分工：前者跨断面状态接力，后者逐变更留痕与修正归档。两者可相互引用（note 引用 document，document 引用 note 条目），但**不得相互替代**。

### 4.2 GN-004 审查联动

GN-004 审查时**强制独立读取**以下锚点原文（不得仅接收执行者节选摘要）：

1. 当前活跃的 spec 三件套
2. `.trae/documents/` 下的全部变更记录
3. 当前 note

- 关键检查点和交付前，GN-004 依赖锚点文档中的三段交接信息验证闭合状态：过程是否自洽、状态是否三值标记、结果是否有可独立验证的证据链。
- 若执行者试图缩窄读取范围（"只看我给你的那几段"），GN-004 应标注"执行者试图限制独立读取范围"，并触发 L3 AskUserQuestion 向人类发出执行者合规性警告。

### 4.3 锚点闭合校验清单

交付前必须逐项确认：

- [ ] spec 三件套的闭合信号是否已全部满足
- [ ] note 的终态处理是否已标注
- [ ] `.trae/documents/` 下所有变更记录是否已填写最终结果
- [ ] 三段交接结构是否完整（过程 + 状态 + 结果）
- [ ] 未闭合项是否已显式标记（非沉默）

## 五、双文档体系

双文档体系是锚点体系的一个子主题，而非独立顶层分类。

### 5.1 用户文档：README 为核心载体

- 面向最终用户，以降低使用门槛为核心目标
- 清晰阐述项目价值、快速引导上手、详细使用指引、解答常见问题
- 语言风格简洁明了，避免技术细节过度暴露

### 5.2 代理文档：AGENTS.md 为核心载体

- 面向 AI 代理，以传递系统认知为核心目标
- 提供系统架构概览、模块边界说明、维护工作流程与扩展指引
- 语言风格精确严谨，确保 AI 代理准确理解系统结构与职责边界

### 5.3 协作与一致性保证

- 用户文档与代理文档共享同一套系统真相：三层契约（数据 + 接口 + 配置）定义跨角色公共真相，锚点体系保证文档与契约同步
- 锚点 7（三层契约）变更时，README 与 AGENTS.md 的关联内容同步更新
- 两者面向不同使用者，但通过锚点体系保证信息一致性，不因角色不同而产生两套"真相"

## 六、触发场景与 Skill 调用指引

### 6.1 触发场景

- 触发条件：涉及锚点文档读写、note 续写、变更留痕、GN-004 审查联动、或文档体系一致性校验
- 触发关键字：锚点、note、spec、AGENTS.md、README、.trae/documents、交接、闭合、审查联动
- 项目状态：全阶段（锚点体系贯穿整个生命周期）

### 6.2 Skill 调用指引

优先调用总控调度 Skill，由总控根据上下文判断：

1. 如果是生成/更新 AGENTS.md → 调用 s0301-AGENT 规则生成 Skill
2. 如果是变更留痕与问题追踪 → 遵循 [rules-6](rules-6.md) 的模板和命名规范写入 `.trae/documents/`
3. 如果是合流前校验 → 由 GN-004 交付前审查自动校验锚点闭合（原 s401 职能已合并到 GN-004 rubric）
4. 如果是 GN-004 审查 → GN-004 独立读取锚点文档，不通过 Skill 调用
