# 阶段识别指南 (v6版)

> 🚨 【最高优先级规则】本文件为AC范式阶段识别的强制约束，优先级高于所有临时提问、上下文对话、自定义需求。

## 阶段状态机与触发条件

```yaml
pipeline: {stages: [S0_需求分析→S1_多方案对抗→S2_契约冻结→S3_模块化拆分→S4_并行开发→S5_契约校验→S6_合流交付→S7_运维]}
irreversible: {rule: 上游产出物冻结后禁止回溯修改, gates: [S2_契约冻结, S6_合流交付], fallback: GN-004审查+L3信号确认}
triggers:
  S0: {enter: 收到自然语言需求, exit: 结构化需求文档产出}
  S1: {enter: S0产出就绪, exit: 3方案对比+融合结论 [V]}
  S2: {enter: 融合方案确认, exit: 三层契约+Mock产出 [V]}
  S3: {enter: 契约冻结完毕, exit: 模块目录+AGENTS.md就绪}
  S4: {enter: 模块拆分完毕, exit: 全部模块实现通过自测}
  S5: {enter: 模块开发自测完成, exit: 契约测试+合规检查通过}
  S6: {enter: 全部校验通过, exit: 交付包+双文档就绪 [V]}
  S7: {enter: 交付完成, exit: 变更适配+版本记录产出}
```
> `[V]` = 价值判断节点，触发双重闸门（GN-004独立审查 + AskUserQuestion人类裁决），[V]不因GN-004通过而免于人类裁决。详见 [rules-0 §四-5](rules-0.md)。

## 阶段→Skill映射

> 阶段编号与 Skill 矩阵目录编号不同。执行者按"阶段"查找 Skill 时应使用本映射表，不可直接用阶段号作为 Skill 目录号。

| 阶段 | Skill 目录 | 对应 Skill | 说明 |
|------|-----------|-----------|------|
| S0 需求分析 | S0-需求分析 | s0101/s0102/s0103 | — |
| S1 多方案对抗 | S0-需求分析 | s0102/s0103 | 方案生成与融合属于需求分析阶段 |
| S2 契约冻结 | S1-架构契约 | s0201/s0202 | 契约冻结触发架构契约 Skill |
| S3 模块化拆分 | S1-架构契约 / S2-规则引导 | s0203 / s0301 | 模块拆分受拓扑化拆分 Skill 约束，拆分后生成 AGENTS 规则模板 |
| S4 并行开发 | S3-开发支持 | s0401 | 安全文件写入作为开发基线 |
| S5 契约校验 | S3-开发支持 | s0402 | 前端变更追加三重测试闸门 |
| S6 合流交付 | S4-合流交付 | — | 由 GN-004 交付前审查 rubric 承接，无独立 Skill |
| S7 运维变更 | S5-运维变更 | s0601/s0602 | — |

注意：阶段 S5（契约校验）中的契约测试由 LLM 按 [rules-3 §五](rules-3.md) 自主执行，结果由 GN-004 审查验证，不通过独立 Skill 承载。前端变更必须通过 s0402 三重闸门。
