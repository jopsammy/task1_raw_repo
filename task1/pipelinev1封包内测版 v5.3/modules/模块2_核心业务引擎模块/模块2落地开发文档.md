## 一、开发背景

### 1.1 开发目标

模块2是整个需求结构化分析工具的核心引擎，需要实现以下功能：

1. **LLM客户端** - 支持多LLM提供商（OpenAI、DeepSeek、豆包）的统一调用
2. **Pipeline编排器** - 协调AC范式的5个阶段执行：
   - 需求锚定与校验
   - 多LLM对抗方案迭代
   - 全局契约生成
   - IDE引导包生成

### 1.2 技术约束

| 约束项 | 说明 |
|--------|------|
| 编程语言 | Python 3.8+ |
| LLM库 | openai (兼容OpenAI格式) |
| 设计模式 | 策略模式、工厂模式、单例模式 |
| 代码风格 | 中文docstring，遵循AC范式规范 |
| 模块依赖 | 模块X（提示词工程模块）、模块1（数据锚点与存储模块） |

### 1.3 时间节点

- **开始时间**：2026-03-01
- **预计完成时间**：2026-03-01
- **状态**：✅ 已完成

### 2.2 类设计

#### LLMProvider（抽象基类）
- 定义LLM提供商的统一接口
- `call()` - 调用LLM API
- `health_check()` - 健康检查

#### OpenAIProvider / DeepSeekProvider / DoubaoProvider
- 继承 LLMProvider
- 实现具体的LLM调用逻辑

#### LLMClient
- 统一管理多个LLM提供商
- 提供降级调用策略
- 健康检查

#### PipelineStage（枚举）
- 定义Pipeline的各个阶段

#### PipelineOrchestrator
- 协调Pipeline各阶段执行
- 日志记录
- 状态管理

## 三、核心实现详解

### 3.1 llm_client.py - LLM客户端

#### 核心类：LLMProvider（抽象基类）

```python
class LLMProvider(ABC):
    @abstractmethod
    def call(self, system_prompt: str, user_prompt: str, **kwargs) -&gt; Dict[str, Any]:
        pass
    
    @abstractmethod
    def health_check(self) -&gt; bool:
        pass
```

**设计思路**：
- 使用策略模式，每个LLM提供商都是独立的策略
- 统一接口，便于扩展新的提供商

#### 核心类：OpenAIProvider（示例）

```python
class OpenAIProvider(LLMProvider):
    def call(self, system_prompt: str, user_prompt: str, **kwargs):
        # 1. 初始化OpenAI客户端
        # 2. 构建messages
        # 3. 调用API，带重试机制
        # 4. 格式化返回结果
```

**关键实现点**：
- 使用openai库（兼容OpenAI格式）
- 指数退避重试策略：2^attempt 秒
- 超时设置300秒（5分钟）
- 健康检查仅发送"hi"消息，避免实际生成

#### 核心类：LLMClient

```python
class LLMClient:
    def call_llm(self, system_prompt, user_prompt, provider=None):
        # 调用指定提供商
    
    def call_llm_with_fallback(self, system_prompt, user_prompt, providers=None):
        # 按优先级尝试多个提供商
```

**关键实现点**：
- 全局单例模式
- 降级策略：按优先级顺序尝试
- 动态添加提供商
- 健康检查所有提供商

### 3.2 pipeline_orchestrator.py - Pipeline编排器

#### 核心类：PipelineOrchestrator

```python
class PipelineOrchestrator:
    def run_requirement_anchoring(self, requirement_text):
        # 阶段1：需求锚定
    
    def run_requirement_validation(self, structured_requirement):
        # 阶段2：需求校验
    
    def run_architecture_iteration(self, validated_requirement):
        # 阶段3：多LLM对抗
    
    def run_contract_generation(self, final_solution):
        # 阶段4：契约生成
    
    def run_ide_bundle_generation(self, final_solution, contracts):
        # 阶段5：IDE引导包
    
    def run_full_pipeline(self, requirement_text):
        # 完整Pipeline
```

**阶段1：需求锚定与结构化**
1. 使用 `requirement_structuring` 模板生成提示词
2. 调用LLM
3. 解析JSON输出

**阶段2：需求校验与修正**
1. 使用 `requirement_validation` 模板
2. 校验并修正结构化需求

**阶段3：多LLM对抗方案迭代**
1. 获取可用的LLM提供商（默认3个）
2. 每个LLM使用不同的模板生成方案（`architecture_solution_llm1/2/3`）
3. 使用 `solution_fusion` 模板融合多个方案

**阶段4：全局契约生成**
1. `interface_contract_gen` - 接口契约
2. `data_contract_gen` - 数据契约
3. `mock_gen` - Mock实现

**阶段5：IDE引导包生成**
1. `agent_md_gen` - 生成agent.md
2. `ide_bundle_gen` - 生成IDE引导包

#### 关键辅助方法：_parse_llm_json_output

```python
def _parse_llm_json_output(self, content: str):
    # 1. 尝试直接解析
    # 2. 失败则提取 ```json ... ``` 块
    # 3. 还是失败则返回原始内容包装
```

**设计思路**：
- LLM输出不一定是纯JSON
- 支持多种格式容错

### 3.3 全局单例模式

两个模块都采用全局单例模式：

```python
# llm_client.py
_llm_client_instance: Optional[LLMClient] = None

def get_llm_client():
    global _llm_client_instance
    if _llm_client_instance is None:
        _llm_client_instance = LLMClient()
    return _llm_client_instance

# pipeline_orchestrator.py
_pipeline_orchestrator_instance: Optional[PipelineOrchestrator] = None

def get_pipeline_orchestrator():
    global _pipeline_orchestrator_instance
    if _pipeline_orchestrator_instance is None:
        _pipeline_orchestrator_instance = PipelineOrchestrator()
    return _pipeline_orchestrator_instance
```

**优点**：
- 避免重复初始化
- 共享状态（如配置、缓存等）
- 统一入口

## 七、常见问题及解决方案

### Q1: LLM调用超时怎么办？

**A**: 
- 检查 `timeout` 配置（默认300秒）
- 检查网络连接
- 使用降级策略 `call_llm_with_fallback()`

### Q2: 如何添加新的LLM提供商？

**A**: 
1. 继承 `LLMProvider`
2. 实现 `call()` 和 `health_check()`
3. 在 `LLMClient._initialize_providers()` 添加初始化
4. 在配置文件添加配置

### Q3: Pipeline执行失败如何调试？

**A**: 
1. 调用 `orchestrator.get_logs()` 查看详细日志
2. 单独执行每个阶段，定位失败点
3. 检查提示词模板是否存在

### Q4: LLM输出不是JSON格式怎么办？

**A**: 
- `_parse_llm_json_output()` 已包含容错逻辑
- 如果持续失败，检查提示词模板
