## 一、定位与意义

### 1.1 系统定位

**结构化需求分析Pipeline系统**是AC范式体系v5.2的**第一块拼图**，也是**AI时代软件工程的入口引擎**。它的核心使命是：

&gt; **将模糊、歧义的自然语言需求，转化为结构化、可校验、可机器处理、符合AC范式规范的工程化输入。**

### 1.2 解决的核心痛点

在AI开发时代，需求分析面临三大核心痛点：

| 痛点 | 传统方案缺陷 | AC范式Pipeline方案 |
|-----|------------|-----------------|
| **歧义性高** | 自然语言需求模糊，不同人理解不同 | 通过LLM结构化+校验，输出无歧义JSON结构 |
| **可追溯性差** | 需求变更无记录，开发过程失控 | 锚点绑定+运行记录，每一步都可追溯 |
| **难以自动化** | 需求文档无法被机器直接理解 | 机读化契约+Schema，可直接驱动后续开发流程 |

### 1.3 适用场景

本Pipeline系统适用于以下场景：

- **新项目启动**：从模糊的需求想法开始，生成完整的工程化方案
- **需求重构**：将现有自然语言需求文档转化为AC范式结构化需求
- **架构探索**：通过三大范式对抗，探索最优架构方案
- **团队协作**：提供统一的需求语言，消除团队理解差异

## 四、Pipeline核心流程详解

本Pipeline包含**5个标准阶段**，每个阶段都有明确的输入、输出和校验机制。

### 4.1 Pipeline阶段总览

| 阶段编号 | 阶段名称 | 核心动作 | 进度占比 |
|---------|---------|---------|---------|
| 阶段1 | 需求锚定 | 自然语言 → 结构化需求 | 10% → 20% |
| 阶段2 | 需求校验 | 校验结构化需求的完整性与合理性 | 20% → 40% |
| 阶段3 | 架构迭代 | 三大范式对抗 + 交叉评审 + 方案融合 | 40% → 70% |
| 阶段4 | 契约生成 | 生成数据契约、接口契约、Mock实现 | 70% → 90% |
| 阶段5 | IDE引导包生成 | 生成agent.md、各模块IDE引导包 | 90% → 100% |

### 4.2 阶段1：需求锚定（Requirement Anchoring）

**目标**：将模糊的自然语言需求转化为结构化的层级化需求树。

**输入**：
- 原始需求文本（自然语言）

**处理**：
1. 调用提示词模板 `requirement_structuring.jinja`
2. LLM将需求拆解为最多5级的层级结构
3. 每个需求条目包含：内容、层级、排序权重、父级ID、标签

**输出**：
- `structured_requirement.json`：结构化需求数据

**数据结构示例**：
```json
{
  "requirements": [
    {
      "content": "用户登录功能",
      "level": 1,
      "order": 0,
      "parent_id": null,
      "tags": ["认证", "核心功能"]
    },
    {
      "content": "支持密码登录",
      "level": 2,
      "order": 0,
      "parent_id": 0,
      "tags": ["登录方式"]
    }
  ]
}
```

### 4.3 阶段2：需求校验（Requirement Validation）

**目标**：校验结构化需求的完整性、一致性、可实现性。

**输入**：
- `structured_requirement.json`（阶段1输出）

**处理**：
1. 调用提示词模板 `requirement_validation.jinja`
2. LLM从多个维度校验需求：
   - 完整性：是否有遗漏的关键功能
   - 一致性：是否有冲突的需求
   - 可实现性：技术上是否可行
   - 清晰性：是否有模糊的描述

**输出**：
- `requirement_validation.json`：校验结果与修正建议

### 4.4 阶段3：架构迭代（Architecture Iteration）

**目标**：通过三大范式对抗生成最优架构方案。

**输入**：
- 校验后的需求数据（阶段2输出）

**处理**：
1. **第一阶段**：三个LLM并行生成方案（每个LLM对应一个软件工程范式）
2. **第二阶段**：对抗评审（每个LLM评审另外两个LLM的方案）
3. **第三阶段**：方案融合（吸收所有方案的优点，生成最终方案）

**输出**：
- `final_architecture.json`：最终融合方案
- 各LLM的独立方案
- 交叉评审意见

（详见第五章「软件工程三大范式对抗机制」）

### 4.5 阶段4：契约生成（Contract Generation）

**目标**：基于最终架构生成三层契约。

**输入**：
- `final_architecture.json`（阶段3输出）

**处理**：
1. 调用 `data_contract_gen.jinja` 生成数据契约（JSON Schema）
2. 调用 `interface_contract_gen.jinja` 生成接口契约（Python存根）
3. 调用 `mock_gen.jinja` 生成预生成Mock实现

**输出**：
- `contracts.json`：包含所有契约的打包文件
- 数据契约（JSON Schema）
- 接口契约（.pyi存根）
- 预生成Mock实现

### 4.6 阶段5：IDE引导包生成（IDE Bundle Generation）

**目标**：生成可直接用于开发的IDE引导包。

**输入**：
- `final_architecture.json`（阶段3输出）
- `contracts.json`（阶段4输出）

**处理**：
1. 调用 `agent_md_gen.jinja` 生成全局AGENT.md
2. 调用 `global_ide_bundle.jinja` 生成全局IDE引导包
3. 调用 `module_ide_bundle.jinja` 为每个模块生成独立引导包
4. 生成引导包索引文件

**输出**：
- `agent.md`：全局AGENT规则文件
- `ide_bundle_global.md`：全局IDE引导包
- `ide_bundle_module_*.md`：各模块IDE引导包
- `ide_bundle_index.md`：引导包索引

## 五、软件工程三大范式对抗机制

这是本Pipeline系统的**核心创新**：通过三个不同软件工程范式的LLM进行对抗性方案生成，避免单视角的设计盲区。

### 5.1 三大范式定义

| 范式编号 | 范式名称 | 核心理念 | 优化目标 |
|---------|---------|---------|---------|
| 范式1 | 边界防腐与高度解耦 | 整洁架构/六边形架构 | 隔离：业务核心与外部依赖绝对分离 |
| 范式2 | 数据流转与状态驱动 | 领域事件/响应式编程 | 流程：单向数据流、事件溯源、状态机 |
| 范式3 | 垂直内聚与直觉实用 | 垂直切片/极限编程 | 实用：按业务Feature组织，YAGNI原则 |

### 5.2 范式1：边界防腐与高度解耦

**核心哲学**：
&gt; 制定严格的接口契约，确保业务核心逻辑与外部依赖（框架、UI、数据库、第三方接口）绝对分离。

**输出特征**：
- 关注点放在 Interface 设计、Adapter 隔离和抽象类的定义上
- 强调控制反转（IoC）和依赖注入（Dependency Injection）
- 即使系统很小，也要考虑未来可以无感替换任意底层组件

### 5.3 范式2：数据流转与状态驱动

**核心哲学**：
&gt; 把整个系统看作是数据的管道，减少共享的可变状态，强调单向数据流、事件溯源或状态机。

**输出特征**：
- 关注点放在事件总线、数据模型、不可变性和管道拦截器的设计上
- 核心是弄清楚"数据从哪来，经过什么变换，到了哪里，触发了什么状态"

### 5.4 范式3：垂直内聚与直觉实用

**核心哲学**：
&gt; 抵制过度抽象，要求按业务 Feature（特性）来组织代码，而不是按技术分层来组织。

**输出特征**：
- 给出非常直接的模块划分，模块之间几乎没有横向的相互依赖
- 关注点是"如何快速增加或删除一个独立业务功能"
- 遵循 YAGNI 原则，用最直白的方式解决问题

### 5.5 对抗评审机制

在三个LLM生成各自方案后，进入**对抗评审环节**：

- LLM1（范式1）评审 LLM2 和 LLM3 的方案
- LLM2（范式2）评审 LLM1 和 LLM3 的方案
- LLM3（范式3）评审 LLM1 和 LLM2 的方案

每个评审都会从自身范式的角度，指出其他方案的缺点和风险。

### 5.6 方案融合

最后，基于三个独立方案和交叉评审意见，生成**融合最终方案**：

- 吸收所有方案的优点
- 规避所有方案的缺点
- 解决三大范式的核心冲突
- 生成兼顾所有约束的全局最优解

## 八、模块0中控能力

模块0（全局调度面板）是本Pipeline系统的**指挥中心**，提供完整的Pipeline控制能力。

### 8.1 Pipeline控制功能

| 功能 | 说明 |
|-----|------|
| **启动** | 启动完整Pipeline（5个阶段） |
| **暂停** | 暂停Pipeline执行（保留当前状态） |
| **继续** | 继续Pipeline执行（从暂停点恢复） |
| **停止** | 停止Pipeline执行（可选择回滚） |
| **回滚** | 回滚到指定阶段，重新执行后续步骤 |

### 8.2 进度监控

模块0实时监控Pipeline的执行状态：

- 当前阶段
- 进度百分比
- 实时日志
- 已完成的阶段结果

### 8.3 运行记录系统

模块0集成了完整的运行记录系统：

- 每次运行生成唯一的 `run_id`
- 记录所有输入、输出、日志、LLM调用信息
- 支持历史记录查询、回顾、复现
- 运行记录保存在 `workspace/runs/` 目录

### 8.4 异常自动处理

模块0内置了常见异常的自动处理逻辑：

- LLM API调用失败 → 自动重试 → 降级到其他提供商
- 模板渲染失败 → 回退到默认模板
- 数据校验失败 → 给出明确的错误提示
- 只有规则未覆盖的异常才触发人工干预

## 九、交付物体系

### 9.1 交付物目录结构

```
workspace/outputs/{project_id}/
├── structured_requirement.json      # 阶段1输出：结构化需求

├── ide_bundle_global.md              # 阶段5输出：全局IDE引导包
├── ide_bundle_module_*.md            # 阶段5输出：各模块IDE引导包
├── ide_bundle_index.md               # 阶段5输出：引导包索引
└── ...（其他交付物）
```

### 9.2 核心交付物说明

| 交付物 | 用途 | 后续阶段使用 |
|------|------|-----------|

| `agent.md` | 全局AGENT规则 | 整个开发周期的最高优先级规则 |
| `ide_bundle_module_*.md` | 模块IDE引导包 | 各模块独立开发 |

## 十、使用指南

### 10.1 环境配置

**安装依赖**：
```bash
cd v1
pip install -r requirements.txt
```

**配置API Key**：
```bash
# 复制配置模板
cp config/config_template.json config/config.json

# 编辑 config.json，填入你的 API Key
```

支持的LLM提供商：
- OpenAI
- DeepSeek
- 豆包（火山引擎）

### 10.2 快速开始（UI方式）

1. 启动UI：`start.bat`
2. 进入「🎯 需求锚定」Tab
3. 输入你的需求文本
4. 点击「启动Pipeline」按钮
5. 等待Pipeline执行完成
6. 在各个Tab查看结果

### 10.3 快速开始（CLI方式）

```bash
# 进入项目目录
cd v1

# 运行Pipeline（直接输入需求文本）
python interfaces/boot.py cli run --text "我要做一个用户登录系统，支持密码登录和短信登录"

# 或者从文件读取需求
python interfaces/boot.py cli run --file my_requirement.txt
```

### 10.4 提示词调试

系统内置了强大的提示词调试面板：

1. 进入「🔧 提示词调试」Tab
2. 选择要调试的提示词模板
3. 编辑模板内容（可选）
4. 填入测试上下文
5. 点击「预览提示词」查看渲染结果
6. 点击「测试运行」实际调用LLM

### 10.5 查看历史记录

所有运行记录都会被保存，可以随时查看和回顾：

- UI方式：进入「📊 运行记录」Tab
- CLI方式：使用 `cli record list` 和 `cli record show` 命令
