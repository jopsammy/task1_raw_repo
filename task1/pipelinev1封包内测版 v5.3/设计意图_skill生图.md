- 一开始已拥有vi标准和标注logo的png
- 用户给出提示词

- 根据这个增强，把原始用户需求、vi标准和这个增强后的用户需求，传入生图api，出图回传
- 把回传的出图，利用Pillow 库把png logo并到左上角或者右上角
- 再整体出图

===实践范例===
# 智能配图模块 · 简化设计手册

> **一句话**：给定文章内容，自动完成"判断哪里需要配图 → 生成画面描述和英文Prompt → 调绘图API出图"的全自动闭环。
>
> **核心思路**：借一个文本LLM做"翻译"（把文章语义翻译成画面描述），再借一个图像模型"画出来"。

---

## 一、核心流水线

```
输入文章 (章节列表)
  │
  ├── 阶段A：文本LLM并行分析每个章节
  │     ① 价值判断 → {"needs_illustration": bool, "reason": str}
  │     ② 筛选出 needs_illustration=true 的章节
  │
  ├── 阶段B：文本LLM并行生成画面描述（仅对筛选出的章节）
  │     → {"description": "中文画面描述", "visual_prompt": "英文绘图Prompt", "tags": [...]}
  │
  └── 阶段C：图像模型并行出图（传入 visual_prompt）
        → PNG文件
```

**设计意图**：两个阶段串行（必须先判断再生成），但每阶段内部并行。不使用同一个模型同时做分析和绘图——文本模型理解语义，图像模型负责产出。

---

## 二、关键架构决策

### 2.1 Provider策略模式（对接绘图API）

绘图API多种多样：有的走Chat Completions接口（如Cherryin代理的Gemini），有的走专用Images Generations接口（如火山方舟Seedream），有的直接返回base64，有的返回URL。

**解法**：定义抽象基类，工厂模式实例化：

```
ImageEngine (编排器，不关心具体是哪个API)
  └── ProviderFactory
        ├── CherryChatCompletionsProvider  (Gemini)
        ├── ArkSeedreamProvider           (火山方舟)
        └── (随时扩展新Provider)
```

每个Provider只需实现两个方法：`generate_image(prompt)` 和 `health_check()`。切换模型只需改配置，零代码改动。

### 2.2 Jinja2模板管理提示词

提示词独立于代码，存为 `.jinja` 模板文件。两个模板就够了：

| 模板 | 用途 | 关键约束 |
|------|------|----------|
| `analyze_value.jinja` | 判断章节是否值得配图 | 核心约束：**"不仅仅是缓解视觉疲劳，必须有助于理解内容"** — 建立硬过滤标准 |
| `describe_scene.jinja` | 生成详细画面描述 | 要求同时产出**中文描述**（给人看）+ **英文Prompt**（给绘图模型用）+ Tags |

### 2.3 中英双语输出策略

让分析LLM同时生成中英文——中文给编辑在UI预览，英文直接喂绘图模型（几乎所有绘图API对英文理解远优于中文）。不需要额外翻译步骤。

### 2.4 并发控制

`asyncio.Semaphore(3)` 限制并发数，既快又不触发API限流。

## 三、提示词设计原则

1. **强制JSON输出**：`请返回JSON格式：{"needs_illustration": boolean, "reason": "..."}` — 让LLM输出程序可解析的结构化数据
2. **硬约束前置**：过滤条件（如"必须有助于理解内容"）放在提示词开头，比放在末尾有效得多
3. **中英分离**：画面描述（Description）用中文，绘图Prompt（Visual Prompt）用英文，各司其职
4. **详细即精准**：Prompt越详细（构图/配色/视觉隐喻），产出的图越匹配预期

---

## 四、踩过的关键坑

| 坑 | 解法 |
|----|------|
| **Windows子线程中asyncio+aiohttp死锁** — 应用卡死 | 所有异步代码在主线程运行，避免子线程中创建事件循环 |
| **绘图API返回格式多变** — URL提取失败 | 实现多层正则降级提取：Markdown格式 → 纯URL → 关键词URL → data字段 → 全局搜索 |
| **健康检查太慢**（实际生图测试每次几十秒） | 健康检查改为极轻量连通性测试（发"hi"，max_tokens=5），不跑完整生成 |
| **JSON字段缺失导致UI崩溃** | 读取数据时防御性校验，缺失字段自动补齐默认值 |
| **API Key硬编码风险** | 用 `api_key_ref` 引用键名间接获取，模板文件清空真实Key |


| 层 | 技术 |
|----|------|
| 异步HTTP + 并发 | `aiohttp` + `asyncio.gather()` + `asyncio.Semaphore` |
| 提示词管理 | `jinja2` 模板引擎 |
| 架构 | 抽象基类 + Provider工厂模式 |
| 数据 | JSON元数据 + 文件系统存图片 |
| 配置 | JSON驱动，`api_key_ref`间接引用密钥 |

## 六、给下一个项目的"最小可行架构"清单

如果你要从零实现"文字→配图"，按这个顺序来：

1. **选一个文本LLM**（分析用）和一个图像API（出图用），两者分开
2. **写两个Jinja2模板**：一个做价值判断 + 一个做画面描述（含中英文）
3. **用 `asyncio.gather()` 并行调文本LLM**：先全量判断价值 → 筛选 → 并行生成描述
4. **用Provider模式封装绘图API**：抽象基类定义 `generate_image()`，至少实现一个Provider
5. **用 `Semaphore` 控制并发数出图**，结果存为 `{id}.png` + JSON元数据
6. **加多层容错**：URL提取降级、字段缺失自动补齐、健康检查极简
7. **注意平台差异**：Windows下别在子线程跑asyncio
